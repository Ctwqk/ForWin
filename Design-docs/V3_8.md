# ForWin V3.8 Observability / Audit Logging 设计规格

更新时间：2026-04-25

来源：`V2_9_3_skill_runtime.md` 当前主干架构 + V3.8 日志系统增量设计。

说明：本文件定义 ForWin 的观测、审计与 LLM trace 增量方案，作为后续实现和审计追溯的版本化依据。V3.8 不从零重建日志系统，而是在现有 `DecisionEvent`、`PromptTrace`、`GenerationTask`、`ArtifactStore` 基础上统一入口、补齐关键链路、增加只读观测 API。V4.5.1 只继续收束后端 trace contract、payload schema 与 redaction 口径；metrics dashboard、SLO 看板和长期运行健康度页面归入 V4.6+。

---

## 1. 设计目标

ForWin 已经不是普通的 `logger.info()` 适用场景。当前系统包含长流程、可回滚 canon 写入、人工检查点、LLM 调用、review / repair、publisher 与 comment ingestion，所以日志必须分层。

V3.8 建立四层观测模型：

- **运行日志**：给开发者排查异常、耗时、调用栈，输出 JSON runtime logs。
- **事件账本**：给系统和前端追踪“发生了什么、为什么发生、谁触发的”，继续使用 `DecisionEvent`。
- **LLM Trace**：记录 prompt 结构、模型、attempts、retry、fallback、失败和输出摘要，继续使用并扩展 `PromptTrace`。
- **Artifact 日志**：大文本、raw prompt / response、context snapshot、frozen candidate 不塞 DB，只存 artifact URI、hash、size、摘要。

核心原则：

- `DecisionEvent` 是有意义的业务 / 审计 / 治理事件账本，不是低级 debug log。
- `PromptTrace` 负责 LLM 可复现信息。
- `ArtifactStore` 负责大对象。
- `GenerationTask` 只保存任务当前视图和阶段简史，不承担完整日志职责。
- 所有重要事件必须绑定 task / project / chapter / stage / causal root。

---

## 2. V3.8 范围边界

本增量覆盖：

- 统一 observability 后端入口
- API runtime task lifecycle 事件
- orchestrator 主链 stage / chapter / review / repair / canon 事件
- LLM attempts / retry / fallback / duration trace
- context assembly / retrieval 摘要事件
- writer output、review verdict、repair、canon commit 摘要
- artifact 保存与只读读取 API
- task timeline、chapter ledger、prompt trace detail 只读 API
- redaction、因果链、失败路径测试

本增量不覆盖：

- 前端日志页面
- 新 DB 表或新列
- publisher / comment sync 全链路事件补齐
- 完整 metrics dashboard
- raw prompt / raw response 内联写入 `DecisionEvent`

V3.8 的实现策略是“后端统一 + 可审计 API 先行”。前端 Task Timeline、Chapter Ledger、LLM Trace Viewer、Failure Explorer 可作为下一增量建立在这些 API 之上。

## 3. Observability 模块

新增内部模块：

```text
forwin/observability/
  context.py
  recorder.py
  events.py
  redaction.py
  logging_config.py
  metrics.py
```

职责：

- `context.py`：定义 `OperationContext`，携带 `event_id`、`causal_root_id`、`parent_event_id`、`task_id`、`project_id`、`arc_id`、`band_id`、`chapter_number`、`stage`、`actor_type`。
- `recorder.py`：定义 `LogRecorder`，统一提供 `record_event()`、`record_stage()`、`record_error()`、`record_llm_attempt()`、`record_artifact()`。
- `events.py`：集中维护 V3.8 新增事件类型和 payload schema helper。
- `redaction.py`：递归脱敏敏感字段。
- `logging_config.py`：配置 JSON runtime logger，输出 stdout，并兼容现有 API file redirect。
- `metrics.py`：提供轻量 duration / count / failure ratio helper，首批只供 payload 使用，不引入外部 metrics 系统。

所有 recorder 写入前必须执行 redaction。所有 recorder 失败不得中断主业务流程，但必须降级写 runtime warning。

### 6.4.1 Stage-aware LLM 分工与 fallback policy

V3.8 后，LLM fallback 不再是一条全局固定链。系统必须按 `task_family`、`stage_key`、`response_format` / `output_schema` 计算 `llm_task_route`，再对当前任务启动时冻结的 profile 候选列表进行重排与过滤。

候选 profile 来源：

```text
本次请求指定 profile 或 runtime 默认 profile
  -> runtime settings 中保存的其他 profiles
  -> .env / environment 派生的 env-managed profiles
  -> 按 api_key + base_url + model 去重
```

profile 类型识别规则：

```text
spark: profile id/name/base_url/model 包含 codex-spark 或 gpt-5.3-codex-spark
kimi:  包含 kimi 或 moonshot
minimax: 包含 minimax 或 minimaxi
openai: 包含 openai 或 gpt-
other: 其他 OpenAI-compatible profile
```

注意：路由只使用已配置 profile，不自动创建 Spark / Kimi / MiniMax profile，也不修改 runtime 默认模型。

V3.8.1 之后，Kimi 与 DeepSeek 的 API key 以 `.env` / environment 为权威来源：

```text
KIMI_API_KEY / KIMI_BASE_URL / KIMI_MODEL
MOONSHOT_API_KEY / MOONSHOT_BASE_URL / MOONSHOT_MODEL  # Kimi alias
DEEPSEEK_API_KEY / DEEPSEEK_BASE_URL / DEEPSEEK_MODEL
```

runtime settings 可以继续保存用户创建的 profile，但如果 `.env` 中配置了同 provider 的 env-managed profile，运行时会用 env profile 替换旧的 Kimi / DeepSeek runtime profile。保存 UI 运行偏好时，env-managed profile 参与返回给前端和任务 fallback，但不会把对应 API key 写回 `data/runtime_settings.json`。

#### 6.4.1.1 路由表

| `llm_task_route` | 匹配任务 / stage | fallback 优先级 | MiniMax 策略 | 审计原因 |
| --- | --- | --- | --- | --- |
| `prose_generation` | `chapter_draft`、`chapter_preview`、`provisional_preview`、`scene_generation`、`scene_stitch`、writer 非 JSON 调用 | Spark -> Kimi -> OpenAI -> other | 默认跳过；只有没有任何合适替代 profile 时才可作为最后兜底 | 正文生成必须稳定、有内容；MiniMax 在真实长链路中不适合作完整写作主力 |
| `repair_generation` | `chapter_rewrite`、repair 正文改写 | Spark -> Kimi -> OpenAI -> other | 默认跳过；只有无替代 profile 时最后兜底 | repair 会影响 canon 接纳，不能把不稳定正文模型放前面 |
| `canon_extraction` | `state_event_extraction`、`thread_time_extraction`、`lore_timeline_notes` | Spark -> Kimi -> OpenAI -> other | 禁止进入正常候选；只有无替代 profile 时最后兜底 | MiniMax 在这些结构化抽取上出现稳定 provider 422 / sensitive 类失败，不能作为 canon 抽取 primary 或 fallback |
| `planning_json` | Genesis JSON、`launch_arc_*`、`arc_plan`、`band_plan`、`chapter_plan`、`scene_breakdown`、其他 JSON planning | Spark -> Kimi -> MiniMax -> OpenAI -> other | 允许作为短 JSON / 低风险规划 fallback | MiniMax 对短 JSON planning、scene breakdown 等通过率可接受，但不能承担完整写作链 |
| `planning_prose` | 非 JSON planning / world model 描述性规划 | Spark -> Kimi -> OpenAI -> other -> MiniMax | 低优先级 | 规划说明可容忍轻微风格差异，但仍优先 Spark / Kimi |
| `review_json` | `chapter_review`、`chapter_review_json_repair`、`repair_verification`、reviewer JSON | Spark -> Kimi -> MiniMax -> OpenAI -> other | 允许作为 review JSON fallback | MiniMax 可用于短审查 JSON，但 review 结论必须进入 PromptTrace 供复盘 |
| `feedback_analysis` | `comment_analysis`、`npc_intents`、`world_pressure`、`phase4` / reader feedback | MiniMax -> Spark -> Kimi -> OpenAI -> other | 优先使用 | 这类任务短、低副作用、适合消耗 MiniMax；失败不应污染 canon |
| `general` | 无法分类的普通调用 | Spark -> Kimi -> OpenAI -> MiniMax -> other | 低优先级 | 未分类调用必须保守，避免 MiniMax 意外进入关键链路 |

#### 6.4.1.2 Retry 与跨模型 fallback 触发条件

单个 profile 内先执行 retry，耗尽后才允许切换到下一个 route profile。跨模型 fallback 只对 transient failure 触发：

```text
408 / 409 / 425 / 429
500 / 502 / 503 / 504 / 529
read timeout / connect timeout / overall timeout
network disconnect / connection reset / remote protocol error
```

以下情况不得触发跨模型 fallback：

```text
400 bad request
401 / 403 auth failure
prompt/schema 本身错误
JSON parse failure
内容质量失败
review fail
governance / checkpoint fail
```

维护备注：fallback 是单个 LLM request 粒度，不是整章失败后再换模型。整章 scene fallback、preview fallback、repair escalation 是业务级 fallback，必须使用独立 stage/event 表达。

#### 6.4.1.3 当前模型职责

- **GPT-5.3-Codex-Spark**：关键链路优先模型。用于正文、scene、repair、canon extraction、planning JSON、review JSON 的 primary 或第一 fallback。
- **Kimi K2.5**：主要中文写作 / 结构化兜底模型。适合正文、规划、抽取；调用时需要 K2.5 兼容参数：禁用 thinking、不发送 temperature、短 stage timeout 至少 120s、小 `max_tokens` 提升到至少 1800。
- **MiniMax M2.7**：短 JSON 与低风险反馈分析模型。适合 `comment_analysis`、`npc_intents`、`world_pressure`、`scene_breakdown`、短 review / planning JSON；不再作为正文、repair、canon extraction 的正常候选。


审计目标：当用户问“最后是哪个模型写成功的”“为什么触发重试”“为什么没有用 MiniMax”“为什么 Kimi / Spark 被选中”时，不需要读 runtime warning，仅通过 `PromptTrace` + `DecisionEvent` 即可回答。

## 7. 只读观测 API

V3.8 新增后端只读 API，不做前端页面。

```text
GET /api/tasks/{task_id}/timeline
GET /api/projects/{project_id}/chapters/{chapter_number}/ledger
GET /api/prompt-traces/{trace_id}
GET /api/artifacts/read?uri=...
```

### 7.1 Task Timeline

返回该 task 的事件时间线：

- stage enter / exit / duration
- chapter progress
- LLM failure / retry / fallback
- review / repair / canon
- checkpoint / pause / terminate
- run completion

### 7.2 Chapter Ledger

返回单章账本：

- chapter plan status
- draft id
- review verdict
- repair attempts
- canon status
- char_count
- issue summary
- frozen candidate URI
- prompt trace ids
- artifact URIs

### 7.3 Prompt Trace Detail

返回指定 trace：

- trace_scope
- stage_key
- template_id / version
- prompt layers metadata
- input snapshot 摘要
- model profile
- attempts
- output summary
- parent trace id
- decision event id

不内联 raw prompt / raw response。

### 7.4 Artifact Read

读取 artifact 时必须限制在 configured artifact store 内，默认只返回截断 preview：

```text
uri
content_type
size
hash
preview
truncated
```

不得允许任意本地路径读取。

## 8. 测试与验收标准

必须新增或扩展测试：

- redaction 递归脱敏 API key、Authorization、cookies、browser session。
- `causal_root_id` 自动回填，`parent_event_id` 可串联 stage / review / repair。
- `run_orchestrator_task()` success / failure / cleanup 都写事件。
- LLM 529 retry、timeout、fallback、最终成功 attempt 都生成 attempts，且不泄露 Authorization。
- preview fallback 成功时，`DecisionEvent` 和 `PromptTrace.attempts_json` 都能证明最终成功模型、profile、attempt 编号和 output chars。
- writer `PromptTrace.attempts_json` 非空，raw prompt / response 只通过 artifact URI 暴露。
- context pruning payload 包含 budget 前后估算和裁剪计数。
- review / repair / canon 失败路径写入标准错误摘要。
- artifact read API 拒绝 artifact root 外 URI。
- live browser E2E 能用 `operation_id` / `test_run_id` 把 UI 操作、API 请求、task、DecisionEvent、PromptTrace 串起来。
- destructive cleanup / project delete 前能保留审计 bundle，删除失败或 SQLite lock 不会掩盖生成结果。

建议首批回归命令：

```bash
pytest tests/test_api_runtime_progress.py tests/test_llm_client_retry.py tests/test_writer_attention_fallback.py tests/test_phase05_regressions.py -q
```

## 9. 2026-04-24 Live / Browser E2E 复盘：日志盲点

本节根据 2026-04-24 对 `http://127.0.0.1:8899` 的 live LLM / browser E2E 测试补充。该轮测试覆盖了真实生成任务、浏览器操作、task timeline API、项目删除 cleanup，并暴露出 V3.8 当前实现与“审计级可追溯”之间的差距。

### 9.1 最终成功模型无法审计级确认

实测现象：

- 正常 writer 链路多次失败。
- runtime log 显示 `MiniMax-M2.7` timeout 后 fallback 到 `kimi-k2.5`。
- Kimi 请求返回 `400 Bad Request`。
- orchestrator 最后进入 preview fallback，并出现 `Writer preview fallback succeeded`。
- 但是成功日志没有记录最终成功 attempt 的 `model` / `profile_id` / `base_url_host`。

结论：

- 只能根据日志顺序推断最终大概率是 MiniMax 的小 token preview fallback 成功。
- 不能审计级证明“最后到底是哪个模型写成的”。

V3.8 补齐要求：

- 每次 LLM 成功 attempt 必须进入 `PromptTrace.attempts_json`，包含 `attempt_group_id`、`attempt_no`、`model`、`profile_id`、`profile_name`、`base_url_host`、`duration_ms`、`input_chars`、`output_chars`、`http_status`。
- `llm_request_succeeded` 的 `DecisionEvent.payload_json` 必须记录 `prompt_trace_id`、`effective_model`、`effective_profile_id`、`attempt_group_id`、`successful_attempt_no`、`fallback_chain`。
- preview fallback、scene fallback、structured extraction fallback 都必须拥有独立 stage key，不能只靠普通 runtime warning。
- `PromptTrace.model_profile_json` 表示“初始候选 profile”；最终成功模型必须通过 `attempts_json` 和 `DecisionEvent` 明确表达。

### 9.2 Preview fallback 不是一等 span

实测现象：

- 正常 writer 失败后，orchestrator 会调用 `write_preview_chapter()` 兜底。
- preview fallback 成功只留下 warning 文本和后续成功结果。
- 该路径没有完整的 started / failed / succeeded span，也没有把原 writer 失败与 fallback 成功用 parent event 串起来。

V3.8 补齐要求：

- 新增或标准化事件：

```text
writer_preview_fallback_started
writer_preview_fallback_attempt_failed
writer_preview_fallback_succeeded
writer_preview_fallback_failed
```

- payload 必须包含：

```text
source_error_class
source_error_message
source_attempt_no
fallback_attempt_no
max_attempts
timeout_seconds
max_tokens
effective_model
prompt_trace_id
duration_ms
output_chars
```

- `writer_preview_fallback_succeeded.parent_event_id` 必须指向触发 fallback 的 `llm_request_failed` 或 writer failure event。

### 9.3 LLM 失败分类不足

实测现象：

- MiniMax timeout、Kimi 400、可能的 429 / 529 / 5xx 都会混在 runtime warning 或 HTTP exception 中。
- 当前日志能看到 status / error message，但缺少统一 error category 和 retryability 判断。

V3.8 补齐要求：

- 所有 LLM failure attempt 增加：

```text
error_category: timeout / rate_limit / provider_overload / bad_request / auth / parse_error / network / unknown
retryable: true/false
fallback_eligible: true/false
final_failure: true/false
provider_request_id
```

- 对 400 类错误必须记录 response 摘要，但不得记录敏感 prompt 或 API key。
- 对 timeout 必须区分 connect timeout、read timeout、overall timeout。

### 9.4 PromptTrace 写入时机存在丢失风险

实测推断：

- `LLMClient` 会收集 attempts。
- `ChapterWriter` 在构造 writer output 的 prompt trace 时 drain attempts。
- 如果调用在构造 writer output 前失败，或 fallback 路径没有保存 trace，attempts 可能只留在 runtime log，无法进入 DB。

V3.8 补齐要求：

- LLM attempt telemetry 必须以 `attempt_group_id` 暂存，并在失败路径也能 flush 到 `PromptTrace` 或 `DecisionEvent`。
- 对“没有 writer output 的失败调用”也要保存 failure prompt trace，`output_summary_json` 标记 `status=failed`。
- caller drain attempts 后必须记录 `drained_attempt_count`，防止多次 drain 导致证据消失。

### 9.5 因果链仍不够稳定

实测风险：

- task lifecycle、writer、review、repair、canon、cleanup 事件已经存在，但不是所有事件都能保证拥有同一个 `causal_root_id`。
- 部分事件的 `parent_event_id` 为空，导致前端 timeline 能按时间看，却不能可靠回答“谁触发了谁”。

V3.8 补齐要求：

- `generation_requested` 或 `continue_requested` 必须成为 task 的 `causal_root_id`。
- `task_operation_started` 必须继承该 root。
- 每个 stage span 的 started / succeeded / failed 形成父子链。
- 每章事件链最小路径：

```text
task_operation_started
  -> chapter_loop_started
  -> context_assembled
  -> chapter_write_started
  -> llm_request_started
  -> llm_request_succeeded / writer_preview_fallback_succeeded
  -> writer_output_built
  -> review_verdict_recorded
  -> canon_commit_started
  -> canon_commit / canon_commit_failed
```

### 9.6 API response envelope 与前端/测试存在不一致

实测现象：

- 某些事件列表 API 返回 `{ "items": [...] }`。
- 测试或前端代码可能期待 `{ "events": [...] }`。
- 这类不一致会让浏览器测试误报“日志失败”，也会让前端页面写兼容逻辑。

V3.8 补齐要求：

- 观测 API response envelope 必须稳定。
- 对事件列表统一使用：

```json
{
  "items": [],
  "next_cursor": "",
  "schema_version": "v3.8"
}
```

- 如果保留 `events` 兼容字段，必须在 schema 中明确为 deprecated alias，并有测试覆盖。

### 9.7 Cleanup / 删除路径会破坏审计证据

实测现象：

- live task 生成完成后，cleanup / project delete 曾遇到 SQLite lock / timeout。
- 后续手动删除项目会移除 DB 证据，导致无法再从数据库验证最终 LLM success attempt。

V3.8 补齐要求：

- destructive action 必须记录：

```text
project_delete_requested
project_delete_started
project_delete_succeeded
project_delete_failed
audit_bundle_exported
```

- 删除项目前默认导出 audit bundle，至少包含 task timeline、chapter ledger、prompt trace summaries、artifact manifest。
- 删除失败必须记录 DB lock / active transaction / retry count / duration。
- cleanup 失败不得覆盖 task 的最终业务状态。

### 9.8 浏览器操作与后端事件缺少强关联

实测现象：

- 浏览器 E2E 能点击 UI 并观察 API。
- 但 UI 操作、API request、后台 task、DecisionEvent、PromptTrace 之间没有统一 `operation_id`。
- 失败报告很难自动定位到对应审计事件。

V3.8 补齐要求：

- 前端发起关键操作时生成或传递 `operation_id`。
- 后端将 `operation_id` 写入 `DecisionEvent.payload_json`、`GenerationTask.stage_history_json` 片段和 runtime log。
- 浏览器测试记录 `test_run_id`，所有 seed、task、publisher job、world studio action 都带 `test_run_id`。
- Failure Explorer 和测试报告可以按 `operation_id` 查询完整链路。

### 9.9 长耗时任务缺少性能归因

实测现象：

- live 章节生成耗时很长。
- 当前能看到部分 duration，但不足以把耗时拆成 queue、context、LLM attempts、review、repair、canon、DB lock、cleanup。

V3.8 补齐要求：

- 每个 stage span 必须记录 `duration_ms`。
- LLM attempts 必须记录 `duration_ms`、`timeout_seconds`、`retry_after`、`sleep_ms`。
- task timeline API 应聚合：

```text
total_duration_ms
llm_duration_ms
retry_sleep_ms
context_duration_ms
review_duration_ms
repair_duration_ms
canon_duration_ms
cleanup_duration_ms
db_lock_wait_ms
```

- 超过阈值的 stage 记录 `performance_warning` event。

### 9.10 Publisher / comment sync 仍是后续盲区

实测重点在生成与浏览器操作，publisher 页面和 extension 状态虽有测试计划，但日志链路还没有达到生成链路同等水位。

V3.8 后续必须补齐：

- upload job claimed / progress / succeeded / failed / cancelled。
- extension heartbeat received / stale / key mismatch。
- browser session synced，但只记录 cookie count、cookie names、current URL、platform id。
- comment sync job created / claimed / succeeded / failed。
- raw comments ingested、duplicate count、signal count、aggregate result。
- 所有 publisher side effect 必须带 `operation_id`、`platform_id`、`job_id`。

### 9.11 Raw prompt / response artifact 覆盖不足

实测问题：

- 发生 provider 400 或 parse failure 时，缺少可审计的 request/response artifact 指针。
- 为避免泄密，不能把完整 prompt / response 放入 `DecisionEvent`；但完全没有 artifact URI 会导致复盘困难。

V3.8 补齐要求：

- 对 live LLM 调用保存 artifact manifest：

```text
raw_prompt_artifact_uri
raw_response_artifact_uri
redacted_request_artifact_uri
response_summary
input_hash
output_hash
```

- 默认只开放 redacted preview；完整 raw artifact 需要本地权限或显式 debug 开关。
- provider 400/parse failure 必须保存 response 摘要 artifact，用于排查 schema mismatch。

### 9.12 最小补齐优先级

为避免 V3.8 看似完成但仍无法审计 live 事故，后续实现优先级调整为：

1. LLM success attempt provenance：最终成功模型、profile、attempt、prompt trace id。
2. Preview fallback 一等 span：started / failed / succeeded 与 parent event。
3. Failure prompt trace flush：失败调用也能进 `PromptTrace`。
4. Causal chain root propagation：task 到 chapter 到 canon 全链路。
5. Operation correlation：browser/API/task/event/trace 统一 `operation_id`。
6. Audit bundle before delete：删除前保留证据。
7. Stage performance aggregation：定位长耗时黑洞。
8. Publisher/comment sync event parity：发布链路达到生成链路同等可审计水位。

## 10. 审计问题覆盖

V3.8 完成后，系统应能通过结构化日志回答：

- 为什么第 23 章被 forced accept？
- 为什么某个角色突然出现或没有出现？
- 为什么 band checkpoint fail / warn？
- 为什么模型 fallback，最终成功模型到底是谁？
- 为什么某章没有使用某个伏笔？
- 为什么 context 超 budget？
- 为什么 canon commit 回滚？
- 哪次 LLM 调用失败、retry、fallback，最终哪次 attempt 成功，耗时多少？
- 哪个 artifact 保存了 raw output / frozen candidate？
- 哪个人工操作 approve / override 了系统决策，理由是什么？
- 哪个浏览器操作触发了某个 API、task、DecisionEvent 和 PromptTrace？
- 为什么删除 / cleanup 失败，是否影响最终生成状态？

这些问题必须通过 `DecisionEvent`、`PromptTrace`、`ArtifactStore` 和只读 API 组合回答，而不是依赖普通 runtime log。

## 13. LLM 路由与可靠性审计补充

### 13.1 Stage-aware fallback chain

V3.8 的 LLM fallback 不是一套全局链条，而是按 `llm_task_route` 分流：

| Route | Stage examples | 默认链条 |
|---|---|---|
| `prose_generation` | `chapter_draft`、`scene_generation`、`scene_stitch` | Spark -> Kimi -> OpenAI / Other |
| `repair_generation` | `chapter_rewrite`、repair stages | Spark -> Kimi -> OpenAI / Other |
| `canon_extraction` | `state_event_extraction`、`thread_time_extraction`、`lore_timeline_notes` | Spark -> Kimi -> OpenAI / DeepSeek / Other |
| `writer_preview` | `writer_preview`、`writer_preview_fallback` | Spark -> Kimi -> OpenAI -> DeepSeek -> MiniMax -> Other |
| `planning_json_low_risk` | `scene_breakdown`、`genesis_brief`、`launch_arc_*` | Spark -> Kimi -> MiniMax -> OpenAI / DeepSeek / Other |
| `planning_json_general` | `world`、`map`、`story_engine`、`book_blueprint`、`bootstrap`、`band_plan`、`chapter_plan` | Spark -> Kimi -> OpenAI / DeepSeek / Other |
| `review_json` | `chapter_review`、`repair_verification` | Spark -> Kimi -> MiniMax -> OpenAI / DeepSeek / Other |
| `feedback_analysis` | `comment_analysis`、`npc_intents`、`world_pressure` | Spark -> Kimi -> MiniMax -> OpenAI / DeepSeek / Other |
| `general` | 未分类调用 | Spark -> Kimi -> OpenAI / DeepSeek / Other |

硬规则：

- Gemini profile 在存在 DeepSeek 或 Kimi 替代时跳过，`skipped_profiles.reason` 分别记录 `replaced_by_deepseek` 或 `replaced_by_kimi`。
- DeepSeek profile 在存在 Kimi 替代时跳过，`skipped_profiles.reason = replaced_by_kimi`。
- MiniMax 禁止进入完整正文、repair、canon/state 抽取、general 与高风险 planning route。
- MiniMax 只保留为 `writer_preview`、低风险 planning JSON、review JSON、feedback analysis 的备选。
- Spark 指 `gpt-5.3-codex-spark`，不是普通 OpenAI-compatible API key profile。ForWin reliability eval 通过本机 Codex App / CLI 的订阅态调用 `codex exec --model gpt-5.3-codex-spark`，profile provider 记为 `codex_cli`，不要求 `.env` 中存在 OpenAI API key。
- Codex CLI eval 默认使用 `--ignore-user-config --ephemeral`。这会保留 Codex 登录 / auth，但不加载用户级 MCP、connector 或本地配置，避免 Figma 等无关 MCP auth 失败污染模型可靠性测试。

### 13.2 Eval 判定必须复用生产 parser

2026-04-24 的 live reliability probe 暴露出一个审计陷阱：MiniMax M2.7 经常在 JSON 前输出 `<think>...</think>`。ForWin 生产路径的 `parse_llm_json()` 会剥离 reasoning 并抽取 JSON 对象；如果 llm-eval 使用裸 `json.loads()`，会把“生产可解析”的响应误判为格式失败。

因此 V3.8 之后的 LLM reliability 测试必须满足：

- JSON case 使用 `parse_llm_json()` 判断 parse/schema。
- `writer_preview` 使用 `ChapterWriter._parse_preview_text()` 判断是否能抽取正文，而不是只检查 `<<FORWIN_BODY>>` 标签。
- raw response 仍必须进入 LLM artifact；即使生产 parser 能修复，也要保留 `<think>`、markdown fence、前后缀文本等原始证据，便于审计模型是否严格遵守契约。
- report 中应区分 `production_parse_ok` 与 `strict_contract_ok`。第一阶段 reliability 以生产可用性为主，后续质量评估再单独看输出契约稳定性。

### 13.3 2026-04-24 位置可靠性结果

使用生产 parser 后，按当前允许位置执行 live probe：

| Profile | Model | Cases | Result | Duration notes |
|---|---|---:|---|---|
| Codex Spark | `gpt-5.3-codex-spark` via Codex CLI | 11 | 11 / 11 pass | 单次约 3.9s-11.6s |
| Kimi | `kimi-k2.5` | 10 | 10 / 10 pass | 单次约 3.4s-39.2s |
| MiniMax | `MiniMax-M2.7` | 8 | 8 / 8 pass | 单次约 5.1s-39.1s；依赖 reasoning 剥离 |
| DeepSeek | `deepseek-chat` | 6 | 6 / 6 pass | 单次约 0.9s-9.9s |

解释：

- Kimi 在 writer preview、canon extraction、低风险 planning、review、feedback analysis 的被测位置可用，但慢调用可接近 40 秒。
- MiniMax 在当前 allowlist 的短任务/低风险任务中可用，但原始输出经常包含 reasoning，不能提升到 canon、repair 或完整正文主链路。
- DeepSeek 在结构化抽取和低风险 JSON 上表现稳定且较快，但当前硬替换策略下，如果 Kimi 可用，DeepSeek 会作为被替换 profile 被跳过；如后续要按实测结果提高 DeepSeek 优先级，需要单独调整路由策略。
- Codex Spark 在 medium direct probes 的全部 stage 上通过，包括 writer preview、scene breakdown、canon extraction、Genesis brief、arc plan、review、feedback analysis。注意：Codex CLI 不适合把泛型 `{"type":"object"}` 强制传给 `--output-schema`，否则会触发 strict schema 400；eval 对 JSON case 使用 prompt 要求 + 生产 parser 校验，只有存在完整 `properties` 的 schema 才传给 Codex CLI。
