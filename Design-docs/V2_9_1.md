## 1. 系统定义

ForWin 是面向长篇连载网文的自动化写作系统。

核心链路：

```text
Arc -> Band -> Chapter -> Scene
Planning -> Provisional -> Writer -> Review -> Repair -> Canon -> Post-acceptance -> Governance/Feedback
```

支持三种运行模式：

| 模式 | 语义 |
| --- | --- |
| `blackbox` | 自动推进；遇到治理阻断、checkpoint warn/fail/error、运行异常时暂停。 |
| `copilot` | 自动写作和修复；关键失败后进入人工 review。 |
| `checkpoint` | 产出后暂停；等待人工审查和批准。 |

---

## 2. 分层架构

| 层 | 来源 | 职责 | 主对象 |
| --- | --- | --- | --- |
| 生产主链 | V2.3 | 写作、结构化抽取、review、repair、canon | `ChapterPlan`、`WriterOutput`、`ChapterReview` |
| 反馈校准层 | V2.6 | 评论信号、聚合、趋势、动作建议 | `CommentSignalCandidate`、`SignalWindowAggregate`、`AudienceHintPack` |
| 体验审查层 | V2.7 | 网文体验计划、WNER、lint、repair patch | `ReaderPromise`、`BandDelightSchedule`、`ReviewContextPack` |
| 治理层 | V2.8.1 | 严格推进、checkpoint、future constraints、审计 | `BandCheckpoint`、`NarrativeConstraint`、`DecisionEvent` |
| SubWorld 控制 | V2.9 | 角色准入、子世界注册表、band 激活与上下文收窄 | `SubWorldSummary`、`SubWorldPlanDelta`、`ChapterEntryTarget` |

SubWorld 不是独立主链，而是附着在 `Arc -> Band -> Chapter` 规划、context 装配、review gate 和 canon gate 上的强约束层。

---

## 3. 规划模型

| 层级 | 职责 |
| --- | --- |
| `Project` | 一本书的根对象。 |
| `Arc` | 长线剧情阶段；也是 `subworld` 的唯一创建者。 |
| `Band` | 3-5 章左右的治理与执行单元；负责激活与 materialize 已规划 cast/slot。 |
| `Chapter` | 正式连载章节。 |
| `Scene` | Writer 内部生成单元。 |
| `SubWorld` | 角色圈层/局部世界注册表；v1 只对 `character` 施加 strict control。 |

Band 仍然是一等治理单元，聚合 band 计划、task contract、provisional、checkpoint、future constraints、next-band compatibility 和 decision events。  
SubWorld 则提供“当前 band/当前章允许谁进入正文与 canon”的角色边界。

## 4. Writer 主链

默认流程：

```text
Scene Breakdown -> Scene Generation -> Scene Stitch -> Structured Extraction
```

保留 single-chapter generation 作为 fallback。

`WriterOutput` 必须包含：

- 正文：`chapter_number`、`title`、`body`、`char_count`、`end_of_chapter_summary`
- scene 结果：`scene_outputs`
- 结构化抽取：`state_changes`、`new_events`、`thread_beats`、`time_advance`、`entity_mentions`
- 运行信息：`generation_meta`

writer 约束：

- 命名人物只能来自 `allowed_entities`
- 若本章允许新人物出场，只能使用 `chapter_entry_targets`
- 无名泛称路人允许出现，但不能固化成 canon 新人物

错误恢复：

- 单模型 retry
- 跨 profile fallback
- parse repair 或 fallback
- `blackbox` 多次 repair 失败后可 forced accept 当前最新版 draft
- forced accept 仍必须经过 band checkpoint

## 5. Review 与 Repair

Review 聚合：

- continuity review
- WNER
- lint signals
- subworld admission

Verdict：

| verdict | 语义 |
| --- | --- |
| `pass` | 允许进入 canon。 |
| `warn` | 可继续，但必须记录 issue。 |
| `fail` | 触发 repair 或人工处理。 |

Repair scope：

```text
scene -> band -> arc
```

运行模式差异：

| 模式 | fail 后行为 |
| --- | --- |
| `checkpoint` | 不自动 rewrite。 |
| `copilot` | 自动修复；耗尽后 `needs_review`。 |
| `blackbox` | 自动修复；耗尽后可 forced accept，但 checkpoint 仍可暂停。 |

Admission guard：

- 任一命名 `character` 不在本章允许名单中，review 直接报错
- 检查来源包括 `entity_mentions`、`state_changes`、`new_events`、`scene_outputs[].involved_entities`

---

## 6. Canon 与 Post-acceptance

只有 accepted chapter 可以写入 canon。

Canon 回写：

- entities
- events
- plot threads
- timeline
- world state snapshots

Post-acceptance：

- memory index upsert
- pacing assessment
- stage analysis
- world pressure update
- NPC intent snapshot
- feedback aggregation

世界状态采用追加式记录，不使用单个大 JSON 覆盖真值。

角色相关的 canon 规则：

- 不再通过 `apply_state_changes()` 自动造人
- 只有已批准实体或已 materialize 的 roster item 可以进入角色真值
- admission guard 失败时，不写 canon，并记录 `CANON_COMMIT_FAILED`

---

## 7. 反馈校准层

支持六类 signal：

- `risk`
- `pacing`
- `confusion`
- `character_heat`
- `relationship_interest`
- `prediction`

数据流：

```text
PublisherRawComment
-> CommentSignalCandidate
-> SignalWindowAggregate
-> ReaderScaleSnapshot
-> score_v1
-> AudienceTrendView
-> ActionMapper
-> AudienceHintPack
```

边界：

- audience feedback 不直接修改 canon
- audience feedback 只作为 planning、pacing、experience、writer hint 的校准信号

## 8. 体验审查层

Experience overlay：

- `ReaderPromise`
- `ArcPayoffMap`
- `BandDelightSchedule`
- `ChapterExperiencePlan`

`BandDelightSchedule` / `ChapterExperiencePlan` 在 V2.9.1 中扩展：

- `active_subworld_ids`
- `chapter_entry_targets`
- `entity_admission_rule`

WNER 使用 `ReviewContextPack`，包含：

- chapter/band/arc overlay
- entity snapshots
- rule snippets
- recent canon events
- recent review notes
- timeline
- world pressure
- audience hints
- lint signals

WNER 输出：

- planned reward tags
- delivered reward tags
- 7 维 experience scores
- evidence-anchored issues
- review notes
- repair instruction

## 9. 治理层

### 9.1 默认策略

新项目和新 blackbox task 默认：

- `progression_mode = serial_canon_band_guard`
- `auto_band_checkpoint = true`
- `manual_checkpoints_enabled = true`
- `future_constraints_enabled = true`

旧项目保持 legacy 兼容。

### 9.2 推进策略

| 策略 | 语义 |
| --- | --- |
| `legacy_relaxed` | 兼容旧行为。 |
| `serial_canon` | 前一章 accepted 后才能写后一章。 |
| `serial_canon_band_guard` | 跨 band 时要求上一 band checkpoint 为 `pass` 或 `overridden`。 |

### 9.3 Band Checkpoint

固定检查：

- band 内章节全部 accepted
- 未处理 chapter review
- latest review fail
- provisional fail
- band task contract
- next-band compatibility
- active future constraints
- future resource preservation

Status：

| status | 行为 |
| --- | --- |
| `pass` | 自动继续。 |
| `warn` | 默认暂停。 |
| `fail` | 暂停，进入 repair 或人工处理。 |
| `error` | 暂停，记录 evaluator error。 |
| `overridden` | 人工放行。 |

### 9.4 Manual Checkpoint

必须包含：

- `boundary_kind`
- `boundary_chapter`
- `reason`

允许 boundary：

- `chapter_start`
- `chapter_accepted`
- `band_end`

### 9.5 Task Contract

使用 JSON contract，不新增任务表。

Task 类型：

- 必须推进
- 必须建立
- 必须埋伏
- 必须保留
- 必须交付体验

### 9.6 Future Constraints

使用 `NarrativeConstraint`。

| level | 语义 |
| --- | --- |
| `hard` | 阻断。 |
| `soft` | warning。 |
| `hint` | info。 |

### 9.7 Issue Group

Review/checkpoint issue 必须归入：

- `fact_conflict`
- `director_imbalance`
- `runtime_observation`
- `governance_action`

## 10. SubWorld 控制

### 10.1 注册表与 bootstrap

新增两张表：

- `sub_worlds`
- `sub_world_roster_items`

强规则：

- 每个项目必须存在一个不可退役的 `global_core`
- 旧项目迁移后，已有 active character 会被挂入 `global_core`
- `ensure_active_arc_resolution()` 前必须先保证 subworld registry 存在

### 10.2 Arc / Rearc 规划

`ArcDirector.plan_arc()` 与 `plan_subworld_delta(...)` 返回 `subworld_delta`：

- `reuse_subworld_ids`
- `retire_subworld_ids`
- `new_subworlds`
- `initial_active_subworld_ids`

约束：

- `Arc` 是 `subworld` 的唯一创建者
- `rearc` 可以复用 / 新建 / 退役 future-facing `subworld`
- `global_core` 允许 reuse，但不允许 retire

### 10.3 Band 激活与 chapter 准入

Band 只负责：

- 激活当前 band 需要的 `subworld`
- materialize 必要的 core cast
- 选择少量 `chapter_entry_targets`

限制：

- 每章最多 1 个 `chapter_entry_target`
- 每个 band 最多 2 个新命名角色 entry target

### 10.4 Context 收窄

`assemble_context()` / `RetrievalBroker` 改为：

- 只读取当前章允许的 entity snapshots
- relations 只保留允许实体相关边
- `active_subworlds`、`allowed_entities`、`chapter_entry_targets` 进入 `ChapterContextPack`

### 10.5 V1 范围

v1 只对 `character` 启用 strict control。  
`location / faction / item / rule` 继续沿用现有平铺世界逻辑。

## 11. 决策日志

`DecisionEvent` 是唯一结构化日志真值。

必须记录：

- generation / continue / run start / run completed
- hard gate hit
- review verdict
- repair chain
- forced accept
- canon commit/fail
- checkpoint create/hit/approve/override
- constraint lifecycle
- governance update
- pause/continue/terminate
- LLM retry/fallback

所有人工动作必须带 reason。

---

## 12. 前端入口

必须提供：

- 项目级治理设置
- 运行级 governance override
- chapter/band manual checkpoint
- task contract 编辑
- constraint 编辑
- blocking reason 决策解释
- checkpoint / review / decision event 跳转
- project detail 中的 `subworlds` 摘要

v1 不要求新增独立 subworld 编辑 UI。

核心 API：

- task contract read/write
- constraint create/update/archive
- manual checkpoint create
- checkpoint approve/override

- decision events
- causal replay
- governance insights
- trope templates
- band experience override

- `ChapterContextPack.allowed_entities`
- `ChapterContextPack.chapter_entry_targets`

## 14. V2.9.1 不包含

- 自动理解所有隐性未来需求
- 完整 188 trope 模板库
- 独立任务表
- 第二套日志真值
- 第二套约束推理引擎
- 完整 scene continuation contract 以外的第二 writer 主链
- 生产级 reader estimate 精准校准
- 完全无 heuristic 的 WNER
- 对 `location / faction / item / rule` 的 strict subworld 控制
- 独立 subworld 编辑后台
