更新时间：2026-04-22

来源：V2.9 系列统一规格 + `V2.9.2` Book Genesis 根层与创建书本前置化实现 + `V2.9.2A` 地区层、跨小世界势力分布、运行时地区草案、文化背景命名体系骨架与文化词库命名生成器实现 + `V2.9.2B` review/repair、task-center 与 API wiring 收口。

## 1. 系统定义

ForWin 是面向长篇连载网文的自动化写作系统。

`V2.9.2` 的核心变化不是改写后续写作主链，而是把“创建书本”升级为**整本书根层真值的诞生地**。

当前系统主对象：

```text
BookGenesis -> Project -> Arc -> Band -> Chapter -> Scene
```

当前运行主链：

```text
Genesis -> Start Writing -> Arc Envelope / Provisional -> Writer -> Review -> Repair -> Canon -> Post-acceptance -> Governance / Feedback
```

不变的原则：

- 叙事层级仍是 `Project -> Arc -> Band -> Chapter -> Scene`
- `Band` 仍是一等治理单元
- 本文件中的旧 `SubWorld` 局部派生控制语义已经降级为历史兼容；当前 `SubWorld` 最终语义是大陆、星球、位面、异世界等大尺度地图容器
- `DecisionEvent` 仍是唯一结构化动作真值

## 2. V2.9.2 相对 V2.9.1 的版本边界

`V2.9.1` 已经完成：

- writer / review / repair / canon 主链
- feedback calibration
- WNER / overlay / lint
- strict governance / checkpoint / constraints / causal replay
- subworld control

`V2.9.2` 新增的是**根层前置化**：

- 新建书本直接进入 Genesis Workflow
- `Project` 根层新增可版本化 `BookGenesisPack`
- Genesis 完成前，不进入章节级生产
- `start-writing` 成为 Genesis 与写作链的唯一交接点
- arc sizing 从“运行时按全书反推”改成“Genesis 先定每个 arc 的 blueprint sizing，再由运行时继承”

`V2.9.2A` 在这个基线之上继续扩展：

- `map_atlas` 从 `submaps + nodes` 升级为 `submaps + regions + nodes`
- Genesis 地图工作台改成 `小世界 / 地区 / 地点` 三类子项
- `world_bible` 新增可复用的 `culture_profiles[]`，承载文化背景与命名体系
- Genesis 命名系统新增 civilization lexicon，可按文化背景为人物名、地区名、地点名、文明别称自动生成候选
- `story_engine` 支持人物多势力归属、势力跨小世界多地区分布
- runtime 新建 `SubWorld` 时同步生成最小地区脚手架，但先作为派生 `region_drafts`

`V2.9.2B` 在这个基线之上继续收口运行语义：

- repair 固定为三次：`scene -> band -> reviewer-decided band/arc`
- `blackbox force accept` 只对第 3 次 rewrite 后仍然 review fail 的真实正文生效
- `GenerationControlInfo` 正式拆分 `accepted / drafted / generated`
- task-center 后端改成 bulk provisional history 匹配，前端轮询改成视图感知
- `api.py` 开始回收为装配层，task-center 能力抽到独立 service

## 3. Project 根层与 Genesis 真值

### 3.1 Project 级状态

`Project` 新增创建状态：

- `creating`
- `genesis_ready`
- `writing`
- `legacy`

语义：

- `creating`：正在 Genesis Workflow 中
- `genesis_ready`：Genesis 已锁定，可以启动写作
- `writing`：已从 Genesis 进入正式写作链
- `legacy`：旧项目，继续沿用旧入口与旧状态语义

### 3.2 BookGenesisRevision

根层真值不是散落字段，而是版本化的 `BookGenesisRevision`。

每个项目持有：

- `active_genesis_revision_id`
- 当前生效 `BookGenesisPack`

当前实现的 `BookGenesisPack` 固定包含：

- `book_brief`
- `world`
- `book_arc_blueprint`
- `subworld_policy`
- `execution_bootstrap`
- `stage_states`

其中 `world` 是唯一世界观根对象，统一收纳：

- `minimum_world_system`
- `minimum_extension_pack`
- `world_bible`
- `map_atlas`
- `story_engine`
- `institution_profiles`
- `resource_economy_profiles`
- `world_extensions`
- `template_libraries`

兼容规则：

- 旧 revision 若仍把 `world_bible / map_atlas / story_engine` 放在顶层，读取时自动升级投影到 `world`
- 一旦 revision 再次写回，只持久化新的 `world` 根结构，不再回写旧顶层键

设计原则：

- Genesis pack 是 **Project 根层真值**
- 后续 arc / band / chapter 只能继承或特化，不能静默推翻根层
- 对根层的修改必须通过显式 revision / decision event 路径发生

### 3.3 Genesis 根地图与关系层

当前实现把 Genesis 世界观统一收口到 `BookGenesisPack.world`，不再把 `world_bible / map_atlas / story_engine` 直接放在 pack 顶层。

`world` 根对象内部的三块核心真值仍然是：

- `world_bible`：文化背景、命名体系、公理、历史切片
- `map_atlas`：从两层地图扩成三层地图
- `story_engine`：从单一描述字段升级为可表达文化命名体系与跨地区、跨势力关系的结构

并新增世界观扩展骨架：

- `minimum_world_system`
- `minimum_extension_pack`
- `institution_profiles`
- `resource_economy_profiles`
- `world_extensions`
- `template_libraries`

设计目标不是新增叙事层级，而是让 Genesis 根层能够稳定表达：

- 文化背景与命名体系
- 小世界与地区树
- 角色的主归属与次归属
- 势力在多个小世界、多个地区的分布
- 制度、资源经济、信仰、信息、生态、审美、秘密与价值冲突等可复用世界观模板

阶段映射也随之固定为：

- `world -> BookGenesisPack.world`
- `map -> BookGenesisPack.world.map_atlas`
- `story_engine -> BookGenesisPack.world.story_engine`

## 4. Genesis Workflow

### 4.1 新建书本入口

`POST /api/projects` 不再只是创建空项目。

新流程固定行为：

1. 创建 `Project`
2. 新流程默认 `operation_mode=blackbox`
3. 创建初始 `BookGenesisRevision`
4. 将项目状态设为 `creating`
5. 返回 `workspace_url`

因此，“创建书本”本身已经成为生产入口，但生产的是**整本书根蓝图**，不是首批章节。

### 4.2 六阶段 Genesis

Genesis v1 固定六阶段：

1. `brief`
2. `world`
3. `map`
4. `story_engine`
5. `book_blueprint`
6. `bootstrap`

每个阶段都支持：

- 编辑
- 生成
- 重生
- 锁定

状态真值写入 `stage_states`，至少包含：

- `stage_key`
- `status`
- `locked`
- `updated_at`
- `last_trace_id`

### 4.3 阶段目标

| 阶段 | 目标 |
| --- | --- |
| `brief` | 定义 premise、题材、受众、情绪和边界 |
| `world` | 定义世界观根模型，包括最小骨架、世界观扩展、文化背景与制度/经济模板 |
| `map` | 定义结构化地图与空间拓扑 |
| `story_engine` | 定义核心关系轴、长期驱动、承诺与代价 |
| `book_blueprint` | 生成全书多 arc 骨架与 arc sizing |
| `bootstrap` | 确认默认治理、subworld policy 和写作启动条件 |

Genesis 完成后，项目进入 `genesis_ready`，默认停住，不自动写章。

### 4.4 Genesis 地图与关系层结构

#### `world`

`BookGenesisPack.world` 是 Genesis 唯一的世界观根对象。

其最小固定结构为：

- `minimum_world_system`
- `minimum_extension_pack`
- `world_bible`
- `map_atlas`
- `story_engine`
- `institution_profiles`
- `resource_economy_profiles`
- `world_extensions`
- `template_libraries`

约束固定为：

- `world` 阶段保存与 AI 改写都以整个 `world` 根对象为真值
- `map` 与 `story_engine` 阶段只是编辑 `world` 下的结构化子切片
- `template_libraries` 用于作者参考与新对象来源，不默认进入 writer / reviewer 长上下文
- 命名生成继续只依赖 `world.world_bible.culture_profiles[]`，不把模板库混入命名来源

#### `world_bible`

`V2.9.2A` 的 `world_bible` 不再只保留总览、规则和命名风格字符串，还新增：

- `culture_profiles[]`

每个 `culture_profile` 至少包含：

- `id`
- `name`
- `summary`
- `inspiration`
- `generator_civilization`
- `generator_overlays`
- `social_markers`
- `aesthetic_keywords`
- `character_name_style`
- `region_name_style`
- `location_name_style`
- `character_name_examples`
- `region_name_examples`
- `location_name_examples`
- `usage_notes`

这使 Genesis 可以先把“文化背景”和“对应命名体系”搭成可引用根对象，后续再把人物名、地区名、地点名逐步填实。

补充规则：

- `generator_civilization` 是命名生成器的主文明，例如 `中华`、`罗马`、`维京`
- `generator_overlays` 是叠加文明，例如 `基督教`、`穆斯林`
- Genesis normalize 会在 `*_name_examples` 为空时，按 `generator_civilization + generator_overlays` 自动补样例
- 前端允许用户继续手工覆盖这些样例；生成器只负责候选，不直接替代最终真值

#### 文化命名生成器

`V2.9.2A` 新增代码内置命名词库与生成器层：

- 支持 `中华 / 维京 / 罗马 / 西欧/英国 / 南美/拉丁 / 基督教 / 穆斯林`
- 支持混合文明，如 `中华+基督教`
- 支持四类输出：
  - `person`
  - `region`
  - `place`
  - `epithet`

命名生成器的职责是：

- 给 `culture_profiles` 生成样例
- 给 Genesis 字段级 UI 提供一键命名候选
- 给 system generation 的 `map / story_engine` prompt 提供命名辅助

命名生成器不直接成为数据库真值；项目内最终采用的名字仍存回 `BookGenesisPack`。

#### `map_atlas`

`V2.9.2A` 的 `map_atlas` 固定包含：

- `submaps`
- `regions`
- `nodes`
- `edges`

其中：

- `submaps`：小世界 / 大舞台
- `regions`：小世界下辖地区
- `nodes`：具体地点

`submaps[]`、`regions[]`、`nodes[]` 现在都要求稳定 `id`；其中 `submaps[]` 与 `nodes[]` 也已在实现里补齐正式引用键，不再只靠名字串联。

`regions[]` 至少包含：

- `id`
- `name`
- `subworld_name`
- `parent_region_id`
- `level`
- `kind`
- `summary`
- `culture_traits`
- `climate`
- `terrain`
- `controller_factions`
- `resource_themes`

地区层规则固定为：

- `level` 只能是 `1` 或 `2`
- `level=1` 时 `parent_region_id` 为空
- `level=2` 时必须指向同一 `subworld_name` 下的 `level=1` 地区
- 不允许第三级嵌套

`nodes[]` 继续保留 `parent_subworld`，并新增：

- `parent_region_id`
- `culture_profile_id`

这使 Genesis 根地图形成：

```text
submap -> region(level 1) -> region(level 2) -> node
```

其中 `node` 仍然是最细粒度地点，不与 `region` 互相替换。

同时：

- `submaps[]` 可选 `culture_profile_id`
- `regions[]` 可选 `culture_profile_id`
- `nodes[]` 可选 `culture_profile_id`

这样地图对象可以显式挂到某个文化背景与命名体系上。

#### `story_engine`

`V2.9.2A` 的 `story_engine` 继续保留旧字段兼容，但新增正式关系字段。

`factions[]` 现在同样要求稳定 `id`，作为跨地图、跨制度模板引用的正式键。

`core_cast[]` 新增：

- `culture_profile_id`
- `home_region`
- `current_region`
- `faction_memberships[]`

`faction_memberships[]` 固定包含：

- `faction_name`
- `relation`
- `rank`
- `is_primary`

约束固定为：

- 允许多个 membership
- 必须且只能有一个 `is_primary=true`

`factions[]` 新增：

- `culture_profile_id`
- `headquarters_region`
- `footprint[]`

`footprint[]` 固定包含：

- `subworld_name`
- `region_id`
- `presence`
- `mode`

`opposition[]` 新增：

- `culture_profile_id`
- `base_region`
- `backing_factions[]`

因此 Genesis 根层现在可以表达：

- 人物家乡位于哪个小世界、哪个地区
- 人物同时属于哪些势力，以及哪个是主归属
- 势力如何在不同小世界、不同地区展开控制或渗透

## 5. Book Arc Blueprint 与 Arc Sizing

### 5.1 全书 Arc 骨架

`book_arc_blueprint.arcs[]` 是整本书 arc 级骨架真值。

每个 arc 至少包含：

- `arc_number`
- `title`
- `arc_synopsis`
- `goal`
- `stakes`
- `payoff_direction`
- `chapter_start`
- `chapter_end`
- `chapter_count`
- `target_size`
- `soft_min`
- `soft_max`

### 5.2 全书总章数的使用边界

`V2.9.2` 的目标是把“全书总章数”限制在 Genesis 阶段使用一次：

- Genesis 生成 `book_arc_blueprint`
- 切出每个 arc 的 `chapter_count`
- 为每个 arc 写入独立 `target_size / soft_min / soft_max`

因此后续运行时不再需要从 `project.target_total_chapters` 反推当前 arc 的 sizing。

### 5.3 ArcPlanVersion 的扩展

写作开始后，从 `book_arc_blueprint` materialize 的 `ArcPlanVersion` 必须持有：

- `arc_number`
- `chapter_start`
- `chapter_end`
- `planned_target_size`
- `planned_soft_min`
- `planned_soft_max`

这使 `ArcPlanVersion` 不再只是 arc synopsis 容器，也成为 arc sizing 的运行时继承点。

## 6. Start Writing 交接

Genesis 与写作主链之间新增显式交接点：

```text
POST /api/projects/{project_id}/start-writing
```

固定行为：

1. 校验 `creation_status == genesis_ready`
2. 读取当前 `active_genesis_revision`
3. materialize 全书 `ArcPlanVersion` 骨架
4. 当前 arc 标记为 `active`，其余 arc 标记为 `planned`
5. **仅为当前 active arc 生成 `ChapterPlan`**
6. 项目状态切到 `writing`
7. 创建写作任务，进入现有 `resolving_arc_envelope -> provisional -> chapter_loop`

边界保持明确：

- Genesis 不生成 `ChapterPlan`
- `provisional` 继续属于写作侧，不属于创建书本侧
- 章节级生产必须从 `start-writing` 之后开始

## 7. Arc Envelope / Provisional / Band Size

`V2.9.2` 不取消现有 `ArcEnvelope` / `provisional` 机制，但改变 sizing 来源。

当前规则：

- `phase24` 对 Genesis-backed 项目优先读取 `ArcPlanVersion.planned_*`
- 只有缺失 persisted sizing 时，才回退旧的“按全书总章数反推”路径

因此新项目 sizing 链路变为：

```text
BookGenesisPack.book_arc_blueprint
-> ArcPlanVersion.planned_target_size / planned_soft_min / planned_soft_max
-> ArcEnvelope.base_target_size / base_soft_min / base_soft_max
-> provisional preview
-> resolved_target_size / detailed_band_size
```

这解决了此前“总章数既像全书输入，又像第一 arc 章数输入”的混用问题。

v1 已覆盖：

- Genesis 六阶段
- `start-writing` 时的 launch arc planning

每条 trace 至少包含：

- `trace_scope`
- `stage_key`
- `template_id`
- `template_version`
- `effective_system_prompt`
- `prompt_layers`
- `input_snapshot`
- `model_profile`
- `attempts`
- `output_summary`
- `decision_event_id`

边界：

- `DecisionEvent` 仍然是结构化动作真值
- `PromptTrace` 是观测快照，不是第二套动作日志

## 9. Genesis Context 继承

后续 planner / writer / reviewer 不再只是“顺便知道一点背景设定”，而是显式继承 Genesis 根层。

当前上下文协议已新增：

- `genesis_context_refs`
- `genesis_world_overview`
- `genesis_map_overview`
- `genesis_story_engine_summary`

这意味着后续 prompt 可以明确知道：

- 自己引用的是哪一版 Genesis
- 当前世界 / 地图 / 长期故事引擎摘要是什么

`V2.9.2` 的要求是：

- Genesis 必须成为后续生产上下文的正式来源
- 根层不是展示性前置文档，而是真实输入

`V2.9.2A` 进一步要求：

- `genesis_map_overview` 必须能够概括 `submaps / regions / nodes`
- 当 legacy active subworld 存在运行时地区草案时，概览里要附带这些局部地区信息
- 运行时 prompt 应优先引用 Genesis 正式地区真值，再补充 legacy runtime region drafts 或已提升的 `map_regions`

## 10. SubWorld 与 Runtime Region Drafts

V4.5 语义更新：本节保留为 V2.9.2 历史兼容说明，不再代表最终地图设计。当前实现已经将 `SubWorld` 升级为大尺度地图容器；城市、宗门外门、遗迹入口等局部舞台必须进入 `map_regions` / `map_nodes`，详见 `map_scheme_c.md`。

- `WorldBible` 是根世界真值
- 旧 `SubWorld` arc 层局部派生能力只保留兼容读取，不再作为新写入目标
- `subworld_policy` 保持 `BookGenesisPack` 顶层，与 `world` 根对象并列，用来声明局部派生规则

当前 policy 至少强调：

- legacy subworld policy 默认只做局部加法
- 必须声明父作用域
- 与根世界冲突时不得静默覆盖

### 10.1 Runtime 新小世界的地区来源

`V2.9.2A` 明确区分两类地区来源：

- Genesis 已知小世界的正式地区：来自 `BookGenesisPack.world.map_atlas.regions[]`
- runtime 新生成小世界的局部地区：来自 `SubWorldPlanItem.region_seeds[]`

这意味着 runtime 仍然可以生成新的 `SubWorld`，但新小世界不能只有名字，必须带最小地区脚手架。

`SubWorldPlanItem.region_seeds[]` 至少支持：

- `name`
- `level`
- `kind`
- `parent_region_name`
- `summary`
- `culture_traits`
- `climate`
- `terrain`
- `controller_factions`

默认规则：

- 至少 1 个 `level=1` 地区
- 可选 0-2 个 `level=2` 地区
- 不允许 `level=3`

### 10.2 Runtime Region Drafts 的落点

V2.9.2 原设计把 runtime 地区草案先落在：

- `SubWorld.metadata_json.region_drafts`
- `SubWorld.metadata_json.region_source = "runtime_generated"`
- `SubWorld.metadata_json.region_promotion_state = "draft"`

V4.5 当前实现已新增正式 SQL 地图表：

- `map_regions`
- `map_region_edges`
- `map_nodes`
- `map_edges`
- `map_generation_runs`

因此新的边界是：

- legacy runtime region drafts 读取时应提升为 `map_regions`，并在 metadata 记录 legacy source。
- 新写入不再把正式地区塞回 `SubWorld.metadata_json.region_drafts`。
- 是否回写 `BookGenesisPack.world.map_atlas.regions[]` 仍必须经过显式 promotion / merge 路径。

新增 root-world 相关治理问题类型：

## 13. V2.9.2B 运行语义收口

### 13.1 Review / Repair

`V2.9.2B` 的 repair 链固定为三次：

1. 第 1 次：`scene`
2. 第 2 次：`band`
3. 第 3 次：reviewer escalation 决定 `band` 或 `arc`

普通 review 阶段只允许给出 `scene / band` repair instruction；不再通过隐式升级逻辑把第 3 次自动推到 `arc`。

`arc` 的语义明确为：

- 调整 arc 指导层 / payoff map
- 重建当前 band 计划
- 重写当前章

它不是一次全局大重排。

专项设计见：

- [V2_9_2_Review_Repair_Strategy.md](/home/taiwei/.codex/worktrees/d41d/ForWin/Design-docs/V2_9_2_Review_Repair_Strategy.md)

### 13.2 Force-Accept 边界

`blackbox` 仍保留“三次不过可以直接放行”的自动化语义，但 force accept 只允许在下面的条件组合下发生：

- 已完成第 3 次 rewrite
- 第 3 次 rewrite 产出了新的 draft
- 新 draft 已重新跑过 review
- 最新 review verdict 仍然是 `fail`

以下情况一律不允许 force accept：

- writer exception
- writer returned none
- repair 没有得到新的可 review draft

这些情况会进入 `needs_review / repair_failed`，而不是静默写入 canon。

### 13.3 Task-Center 统计口径

`V2.9.2B` 正式把 task-center 的章节统计拆成三套：

- `accepted_chapters`：只表示已入 canon
- `drafted_chapters`：有 draft 但未 accepted
- `generated_chapters`：`drafted + needs_review + accepted`

前端展示规则也同步调整为：

- “已生成”看 `generated_chapters`
- “已接受”看 `accepted_chapters`
- review 阻塞看 `pending_review_chapters`

因此，`drafted` 不再被混算成 `accepted / completed`。

### 13.4 Task-Center 性能与 API 收口

本轮没有重写整个 API 层，但先收口了 task-center 这块最高耦合区域：

- 后端把 provisional history 的 per-task 查询改成 bulk 匹配
- 前端 task-center 轮询不再每 2.5 秒顺带刷新整份 books 列表
- 首页平台状态改成视图感知轮询，不在默认书本页后台持续打接口
- `api.py` 通过 `ApiRouteDeps` 接线，task-center 聚合逻辑抽到独立 `TaskCenterService`

这一步的目标不是“一次性拆完全部巨型文件”，而是先把 task-center 的语义错误、N+1 和 API 大总线问题收住。
