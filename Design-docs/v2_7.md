# v2.7 增量更新计划：Web-Novel Experience Layer + Evidence-Anchored WNER

**摘要**
- 以当前分支已接入的 `experience overlay` 骨架、`HistoricalReviewHub`、rewrite loop、`AudienceHintPack` 为基线，本次不改 v2.3 主链：`WritingOrchestrator -> Writer -> Review Hub -> State Updater`。
- 这次增量的目标是把 v2.7 从“字段占位 + 启发式 reviewer”升级为“可调度的网文体验规划层 + 有证据的体验 reviewer + 与 v2.6 校准层联动”，同时保留已有三种模式语义。
- 五个问题按验收项处理：WNER 改为 LLM-first 证据审查；band cadence 从“计划合法”提升到“计划 + 实际交付”双重判断；continuity/webnovel repair 继续合并；rewrite 异常继续纳入模式化翻工；lint 工具从空壳变成真实信号源。

**实现变更**
- `Overlay 类型扩展`：在现有 JSON 持久化结构上做向后兼容扩展，不新增主表。`ReaderPromise` 增加 `genre_promise`、`core_pleasures`、`ambiguity_mode`、`world_legibility_target`；`ArcPayoffMap` 增加 `revelation_layers`、`ambiguity_constraints`；`BandDelightSchedule` 增加 `curiosity_beats` 与 `ambiguity_payoffs`；`ChapterExperiencePlan` 增加 `question_hook`、`question_resolution`、`rule_anchors`、`relationship_or_status_shift`、`minimum_progress_channels`。
- `Trope library`：保留当前 `trope_library` 作为运行时 registry 入口，不上数据库。把 starter 数据改为版本化 JSON 种子并通过 loader 读入；模板 schema 固定为 `template_id / display_name / category / setup_requirement / payoff_shape / risk_flags / best_window / recommended_hook_types`。首版内置覆盖五类 reward 的 starter pack，并提供后续导入完整 188 模板的同构导入口。
- `Arc/near-term planning`：继续复用 `ArcDirector.draft_arc_structure` 的单次结构草案调用，不增加新的 planning LLM call。Arc 结构输出必须显式给出 `reader_promise`、`macro_payoffs`、`revelation_layers`、`ambiguity_constraints`；`ArcEnvelopeManager` 负责确定性派生 band 与 chapter 体验计划。
- `Band cadence 规则`：每个 3-5 章 band 默认保证 `1-2` 个 `power/progress`、`1` 个 `social`、`1` 个 `mystery`，再按 arc 宏观 payoff 决定补 `justice` 或 `emotion`；必须有一个 `immersion anchor scene`；必须有一组“至少 resolve 一个问题并 open 一个更好的问题”的 `curiosity_beats`；`stall_guard_max_gap` 只用来补空挡，不允许覆盖掉基础 mix。
- `Managed ambiguity`：band 级显式排 `confirmation / reversal / constraint` 三类 `ambiguity_payoffs`。如果 arc 设定为高 ambiguity，planner 仍必须给出“哪些规则不能被打破”的 constraints；WNER 只接受“受控的不确定”，不接受随机失真。
- `Writer 保持原预算`：scene breakdown 继续只扩 `micro_hook + reward_beat_tag + immersion_anchor + progress_marker`，不再加新的 writer-side LLM call。fallback scene 必须从 `ChapterExperiencePlan` 继承这些字段。Writer 仍只读 `AudienceHintView`，不读原始评论，不读 review notes。
- `ReviewContextBuilder`：新增 reviewer 专用 `ReviewContextPack`，不要继续拿 `ChapterContextPack` 硬顶。它必须包含：本章/本 band/本 arc 的 overlay；当前章相关实体快照；`kind == rule` 的 active rule entities 与最近规则变更；最近 5 条按“实体重叠 + thread 重叠 + 时间新近度”排序的 canon events；当前 band 内近几章的 review meta 摘要；timeline、world pressure、audience hints、confirmed signals、lint signals。
- `WNER 主审查器`：改为 LLM-first JSON reviewer，复用现有 `llm_client` 和 `parse_llm_json` 重试范式。一次 chapter 默认只加 `1` 次 review LLM call，最多 `2` 次 parse/repair 重试；失败后降级到当前 heuristic WNER-lite，不能因 WNER parse 失败直接卡死章节。
- `WNER 输出契约`：必须输出 `planned_reward_tags`、`delivered_reward_tags`、`experience_scores`、`issues`、`review_notes`、`repair_instruction`。`experience_scores` 固定包含 `narrative_understanding / attentional_focus / emotional_engagement / narrative_presence / payoff_delivery / stall_tolerance / hook_efficiency`。所有 `warn/fail` 级 issue 必须引用 `ReviewContextPack` 中的证据 id，不能给无锚定结论。
- `WNER 判定规则`：`no-payoff stall` 只在“当前 band 内连续章节没有 reward 交付，且 question/progress/status/rule consistency 也都没推进”时判 `fail`；“拖但有问题梯子/微进展/关系变化/规则稳态”的章节只能 `warn` 或 `pass`。`immersion rupture` 只在因果、 stakes、规则可读性出现断裂并能指出冲突证据时判 `fail`。
- `Lint 真实接线`：把现有 `LintReviewer` 改为 `LintSignalCollector`。Vale、textlint、LanguageTool、reviewdog 全部作为“信号源”接入，输出统一 `LintSignal`；WNER 决定这些信号是否升级为 issue。工具缺失时返回空信号，不改变主链行为。
- `Review hub 聚合`：保持 continuity / lint / WNER 三者分离，最终仍由 `HistoricalReviewHub` 聚合 verdict；repair instruction 继续做 merged patch，并把 `review_notes` 一并写入 `review_meta_json`，不新建 review notes 表。
- `Rewrite 机制`：checkpoint 仍不自动翻工；copilot / blackbox 在 `verdict == fail` 时都必须“改低层设计再重写章节”。第 1 次尝试按 merged `repair_scope` 执行；若同类失败持续存在，再按 `scene -> band -> arc` 升级。band/arc patch 必须先重生成受影响的 `BandDelightSchedule` 与 `ChapterExperiencePlan` 后再 rewrite。rewrite 异常和 `writer_output is None` 继续算失败次数。
- `三种模式`：checkpoint 只展示证据与 patch 建议，不自动 rewrite；copilot 三次失败后 `needs_review`；blackbox 三次失败后保持“强行通过当前最新版 draft”，不转人工。这一语义保持不变，且必须覆盖 rewrite 自身异常路径。
- `v2.6 完整 7C 接入`：复用现有 `ActionMapper`/`AudienceHintPack`，不重造一套 audience 子系统。新增 experience-aware actions：`boost_reward_density`、`clarify_rule_legibility`、`repair_immersion`、`protect_character_heat`、`hold_managed_ambiguity`。`Phase24` 只把 `AudienceHintPack + long-window trends` 当作校准器：`pacing dryness` 缩短 reward gap，`confusion/risk` 增加 rule anchors 与 clarification beats，`character_heat` 提高 social/emotion 模板权重。WNER 同样只把这些信号当 audience confirmation，不当真值来源。

**接口与类型**
- `内部协议`：新增 `ReviewContextPack`、`ReviewNote`、`LintSignal`、`CuriosityBeat`、`AmbiguityPayoff`；扩展 `ReaderPromise / ArcPayoffMap / BandDelightSchedule / ChapterExperiencePlan / ReviewVerdict`，所有新增字段默认空值并保持旧 JSON 可读。
- `ReviewVerdict`：保留现有字段，扩展 `review_notes`；`experience_scores` 从当前 4 维扩成 7 维但继续沿用同一个字段名；`repair_instruction.design_patch` 固定允许 chapter/band/arc 三级 overlay patch。
- `ChapterReviewDetail API`：新增 `review_notes`、`lint_signals`、`evidence_refs`、`proposed_design_patch`，并把扩展后的 `experience_scores` 原样暴露；checkpoint / copilot UI 依赖这些字段展示“计划奖励 vs 实际奖励”“问题梯子/规则锚点/推荐 patch”。
- `ProjectSummary / ProjectDetail API`：在现有 `active_reader_promise`、`active_band_reward_mix`、`active_band_stall_guard` 基础上，新增 `active_revelation_layers`、`active_band_curiosity_beats`、`active_band_template_ids`。
- `Co-pilot API`：新增只读 `GET /api/tropes/templates` 返回 starter trope registry；新增 `POST /api/projects/{project_id}/bands/{band_id}/experience` 允许人工在 co-pilot 模式覆盖 `scheduled_rewards / curiosity_beats / immersion_anchor_scene_goal`，服务端持久化后必须重生成该 band 的 chapter experience plans。

**测试计划**
- `Overlay 派生`：验证 arc 结构草案能稳定生成 `reader_promise / arc_payoff_map / revelation_layers / ambiguity_constraints`；band 计划满足基础 reward mix、stall guard、immersion anchor、question ladder。
- `Trope registry`：验证 starter pack 覆盖五类 reward；JSON loader 与未来导入格式同构；非法模板被拒绝且不会破坏默认 registry。
- `Writer 预算回归`：验证 scene breakdown/output 只扩既有 contract，不新增 writer-side LLM call；fallback scene 正确继承 experience fields。
- `WNER 主路径`：验证 LLM reviewer 能输出 7 维分数、带证据 issue、review notes、repair instruction；“低 payoff 但有问题梯子/进展”的章节不误判为 fail；“规则断裂/因果断裂”能基于证据 fail。
- `WNER 降级`：验证 parse 失败、LLM 超时、工具缺失时回退到 heuristic WNER-lite + continuity，不因 reviewer 子模块故障破坏主流程。
- `Band stall`：验证判断基于“当前 band 最近章节实际 review meta + canon 证据”，而不是只看 schedule 合法性。
- `Lint integration`：验证 Vale/textlint/LanguageTool/reviewdog 的输出进入 `LintSignal`，并且只有在 WNER 采纳时才形成 issue。
- `翻工闭环`：保留并扩展现有回归，覆盖 checkpoint 不 rewrite、copilot 三次 fail 后暂停、blackbox 三次 fail 后强过、rewrite 异常照样走模式语义、merged repair instruction 不丢 continuity。
- `7C 校准`：验证 confirmed `pacing/confusion/risk/character_heat` 能影响下一 band 的 overlay 与 WNER urgency；无 audience signal 时系统回到 plan-first 行为。
- `API/兼容`：验证旧 review_meta_json、旧 experience_plan_json、旧 band schedule rows 都能正常读取；新增 API 字段为空时不影响现有页面。

**假设与默认值**
- 设计真值以 `v2.3` 与 `v2.6` 文档为准，`review-design.rtf` 只在不冲突时借用命名和 reviewer 组织方式，不作为强制规格。
- 这次只落 `starter trope library + 导入口`，不在本次增量里人工完整录入全部 188 模板；但导入格式必须与 starter schema 完全一致，后续扩库不再改代码接口。
- WNER 使用现有同一套 `llm_client` 与模型配置，不单独引入 review 模型；默认接受每章 `+1` 次 reviewer LLM call 的成本。
- `rule snippets` 的第一来源是 `kind == rule` 的实体与其最近状态/事件证据；如果项目当前没有 rule 实体，再降级使用 `revelation_layers / ambiguity_constraints / awe_kit` 作为 reviewer 的规则背景。
- 不新增 trope library 表、不新增 review notes 表；优先复用现有 JSON 列与现有 audience feedback 表，只有当查询成本成为瓶颈时再做表提升。
