# ForWin V3.8 Observability / Audit Logging 设计规格

更新时间：2026-04-24

来源：`V2_9_3_skill_runtime.md` 当前主干架构 + V3.8 日志系统增量设计。

说明：本文件定义 ForWin 的观测、审计与 LLM trace 增量方案，作为后续实现和审计追溯的版本化依据。V3.8 不从零重建日志系统，而是在现有 `DecisionEvent`、`PromptTrace`、`GenerationTask`、`ArtifactStore` 基础上统一入口、补齐关键链路、增加只读观测 API。

---

## 1. 设计目标

ForWin 已经不是普通的 `logger.info()` 适用场景。当前系统包含长流程、可回滚 canon 写入、人工检查点、LLM 调用、review / repair、publisher 与 comment ingestion，所以日志必须分层。

V3.8 建立四层观测模型：

- **运行日志**：给开发者排查异常、耗时、调用栈，输出 JSON runtime logs。
- **事件账本**：给系统和前端追踪“发生了什么、为什么发生、谁触发的”，继续使用 `DecisionEvent`。
- **LLM Trace**：记录 prompt 结构、模型、attempts、retry、fallback、失败和输出摘要，继续使用并扩展 `PromptTrace`。
- **Artifact 日志**：大文本、raw prompt / response、context snapshot、frozen candidate 不塞 DB，只存 artifact URI、hash、size、摘要。

核心原则：

- `DecisionEvent` 是有意义的业务 / 审计 / 治理事件账本，不是低级 debug log。
- `PromptTrace` 负责 LLM 可复现信息。
- `ArtifactStore` 负责大对象。
- `GenerationTask` 只保存任务当前视图和阶段简史，不承担完整日志职责。
- 所有重要事件必须绑定 task / project / chapter / stage / causal root。

---

## 2. V3.8 范围边界

本增量覆盖：

- 统一 observability 后端入口
- API runtime task lifecycle 事件
- orchestrator 主链 stage / chapter / review / repair / canon 事件
- LLM attempts / retry / fallback / duration trace
- context assembly / retrieval 摘要事件
- writer output、review verdict、repair、canon commit 摘要
- artifact 保存与只读读取 API
- task timeline、chapter ledger、prompt trace detail 只读 API
- redaction、因果链、失败路径测试

本增量不覆盖：

- 前端日志页面
- 新 DB 表或新列
- publisher / comment sync 全链路事件补齐
- 完整 metrics dashboard
- raw prompt / raw response 内联写入 `DecisionEvent`

V3.8 的实现策略是“后端统一 + 可审计 API 先行”。前端 Task Timeline、Chapter Ledger、LLM Trace Viewer、Failure Explorer 可作为下一增量建立在这些 API 之上。

## 3. Observability 模块

新增内部模块：

```text
forwin/observability/
  context.py
  recorder.py
  events.py
  redaction.py
  logging_config.py
  metrics.py
```

职责：

- `context.py`：定义 `OperationContext`，携带 `event_id`、`causal_root_id`、`parent_event_id`、`task_id`、`project_id`、`arc_id`、`band_id`、`chapter_number`、`stage`、`actor_type`。
- `recorder.py`：定义 `LogRecorder`，统一提供 `record_event()`、`record_stage()`、`record_error()`、`record_llm_attempt()`、`record_artifact()`。
- `events.py`：集中维护 V3.8 新增事件类型和 payload schema helper。
- `redaction.py`：递归脱敏敏感字段。
- `logging_config.py`：配置 JSON runtime logger，输出 stdout，并兼容现有 API file redirect。
- `metrics.py`：提供轻量 duration / count / failure ratio helper，首批只供 payload 使用，不引入外部 metrics 系统。

所有 recorder 写入前必须执行 redaction。所有 recorder 失败不得中断主业务流程，但必须降级写 runtime warning。

## 4. 标准事件字段

所有重要事件至少应具备以下信息。DB 中已有列优先落已有列，其他字段进入 `payload_json`。

```text
event_id
causal_root_id
parent_event_id
task_id
project_id
arc_id
band_id
chapter_number
stage
event_family
event_type
actor_type
summary
reason
payload_json
related_object_type
related_object_id
duration_ms
status_before
status_after
error_class
error_message
stack_hash
```

敏感字段必须脱敏：

```text
api_key
Authorization
cookies
publisher 登录信息
raw browser session
完整用户评论作者信息
完整 LLM prompt / response
```

完整 prompt / response 默认只允许写入 `ArtifactStore`，并通过 URI 与摘要关联回 `PromptTrace`。


## 6. 首批接入点

### 6.1 API Runtime

接入位置：

```text
forwin/api_runtime.py
forwin/api.py
forwin/api_task_routes.py
forwin/api_project_ops.py
forwin/api_governance_ops.py
```

必须记录：

- 创建生成任务
- 继续生成任务
- task operation started / succeeded / failed
- cleanup started / finished
- pause / terminate request
- manual approve review
- checkpoint override
- governance update
- runtime settings update，且只记录变更项，不记录 API key

`run_orchestrator_task()` 是首批必改点。它目前主要在异常时写 runtime logger 并更新 task 状态，V3.8 要补齐任务生命周期事件。

### 6.2 Orchestrator

接入位置：

```text
forwin/orchestrator/loop.py
```

这是最高优先级链路。必须把以下阶段变成可追踪 span：

```text
planning_arc
creating_project
seeding_state
resolving_arc_envelope
provisional_gate
chapter_loop
context_assembly
chapter_write
continuity_check
historical_review
repair_loop
canon_commit
memory_index_upsert
band_checkpoint
feedback_aggregation
run_completed
```

每个 span 至少记录：

- started / succeeded / failed
- duration_ms
- input summary
- output summary
- related artifact URI

### 6.3 Context / Retrieval

接入位置：

```text
forwin/context/assembler.py
forwin/retrieval/broker.py
```

必须记录：

- `context_assembled`
- `context_pruned`
- `memory_search_started`
- `memory_search_succeeded`
- `memory_search_failed`

关键 payload：

```text
active_entities_count

summaries_count
memories_count
constraints_count

audience_hints_present
estimated_context_chars_before
estimated_context_chars_after

pruned_relations
retrieved_memory_chapters
```

这组事件用于回答：为什么某角色没出现、为什么 writer 不知道伏笔、为什么上下文超 budget。

### 6.4 LLM Client

接入位置：

```text
forwin/writer/llm_client.py
```

必须记录：

- `llm_request_started`
- `llm_request_succeeded`
- `llm_request_failed`
- `retry_attempt`
- `fallback_profile_switched`
- `llm_response_parse_failed`

LLM client 只收集 attempt telemetry，不直接写业务事件。最终由 caller 或 recorder 绑定到 `PromptTrace` 与 `DecisionEvent`。

### 6.5 ChapterWriter

接入位置：

```text
forwin/writer/chapter_writer.py
```

必须记录：

- chapter write started
- scene breakdown started / succeeded / failed
- scene generation started / succeeded / failed
- scene stitch started / succeeded / failed
- structured extraction started
- structured extraction degraded
- writer output built
- scene fallback used
- writer output artifact saved

```text

scene_count

char_count
state_changes_count
events_count
thread_beats_count
lore_candidates_count

structured_extraction_status
model_fallbacks

immersion_anchors

```

### 6.6 Review / Repair / Canon

接入位置：

```text
forwin/reviewer/*
forwin/checker/rules.py
forwin/orchestrator/loop.py
forwin/state/updater.py
```

必须记录：

- review started
- review verdict recorded
- repair started / failed / succeeded
- repair scope escalation
- forced accept applied
- canon commit started / succeeded / failed
- transaction rollback
- frozen candidate saved

repair 日志必须能看出策略升级：

```text
第 1 次：只修正文
第 2 次：允许改 chapter plan
第 3 次：允许改 band plan
第 4 次：forced accept
```

canon commit 成功记录 state changes、events、thread beats、timeline advance、新实体/关系计数。失败时记录 error 摘要、stack hash、frozen artifact URI。

## 7. 只读观测 API

V3.8 新增后端只读 API，不做前端页面。

```text
GET /api/tasks/{task_id}/timeline
GET /api/projects/{project_id}/chapters/{chapter_number}/ledger
GET /api/prompt-traces/{trace_id}
GET /api/artifacts/read?uri=...
```

### 7.1 Task Timeline

返回该 task 的事件时间线：

- stage enter / exit / duration
- chapter progress
- LLM failure / retry / fallback
- review / repair / canon
- checkpoint / pause / terminate
- run completion

### 7.2 Chapter Ledger

返回单章账本：

- chapter plan status
- draft id
- review verdict
- repair attempts
- canon status
- char_count
- issue summary
- frozen candidate URI
- prompt trace ids
- artifact URIs

### 7.3 Prompt Trace Detail

返回指定 trace：

- trace_scope
- stage_key
- template_id / version
- prompt layers metadata
- input snapshot 摘要
- model profile
- attempts
- output summary
- parent trace id
- decision event id

不内联 raw prompt / raw response。

### 7.4 Artifact Read

读取 artifact 时必须限制在 configured artifact store 内，默认只返回截断 preview：

```text
uri
content_type
size
hash
preview
truncated
```

不得允许任意本地路径读取。

## 8. 测试与验收标准

必须新增或扩展测试：

- redaction 递归脱敏 API key、Authorization、cookies、browser session。
- `causal_root_id` 自动回填，`parent_event_id` 可串联 stage / review / repair。
- `run_orchestrator_task()` success / failure / cleanup 都写事件。
- LLM 529 retry、timeout、fallback 生成 attempts，且不泄露 Authorization。
- writer `PromptTrace.attempts_json` 非空，raw prompt / response 只通过 artifact URI 暴露。
- context pruning payload 包含 budget 前后估算和裁剪计数。
- review / repair / canon 失败路径写入标准错误摘要。
- artifact read API 拒绝 artifact root 外 URI。

建议首批回归命令：

```bash
pytest tests/test_api_runtime_progress.py tests/test_llm_client_retry.py tests/test_writer_attention_fallback.py tests/test_phase05_regressions.py -q
```

## 9. 审计问题覆盖

V3.8 完成后，系统应能通过结构化日志回答：

- 为什么第 23 章被 forced accept？
- 为什么某个角色突然出现或没有出现？
- 为什么 band checkpoint fail / warn？
- 为什么模型 fallback？
- 为什么某章没有使用某个伏笔？
- 为什么 context 超 budget？
- 为什么 canon commit 回滚？
- 哪次 LLM 调用失败、retry、fallback，耗时多少？
- 哪个 artifact 保存了 raw output / frozen candidate？
- 哪个人工操作 approve / override 了系统决策，理由是什么？

这些问题必须通过 `DecisionEvent`、`PromptTrace`、`ArtifactStore` 和只读 API 组合回答，而不是依赖普通 runtime log。

V3.8 之后建议拆分：

7. metrics dashboard 与长期运行健康度统计。

## 11. 版本决策摘要

V3.8 的关键取舍：

- 不把 `DecisionEvent` 变成 debug log。
- 不把完整 prompt / response / 正文塞进 DB。
- 不新增 schema，先用标准 payload 完成可观测闭环。
- 不先做前端，先保证后端事件链和只读 API 可审计。
- 所有事件必须能追溯到 task、project、chapter、stage、parent event、causal root。
