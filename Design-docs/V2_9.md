# Full-Rollout SubWorld Control

说明：本文件保留为 `V2.9` 的专项设计稿。当前主干已将 `V2.8.1` 与 `V2.9` 合并并前推为统一规格 [V2_9_2.md](/home/taiwei/ForWin/Design-docs/V2_9_2.md)。

版本边界说明：`subworld` 继续归属 `V2.9` 规格。当前主干已经包含本规格的前置落地能力；`V2.8.1` 的正式收口不以是否继续扩展 `subworld` 为标准，也不要求回退这些已落地能力。

## Summary
实现目标是把“writer 写到谁，谁就进世界”的链路改成受控链路：

- `Arc` 是 `subworld` 的唯一创建者
- 首个 active arc 生成初始 `subworld`
- 后续 `rearc` 也可以新增 / 复用 / 退役 `subworld`
- `Band` 只负责激活和 materialize 已规划的 cast/slot
- `Chapter/Writer` 只能消费当前 band 允许的人物
- 所有项目全量切换到新机制，不保留 auto-create 新人物的 canon 路径

v1 只把 `character` 纳入 strict subworld 控制；`location / faction / item / rule` 继续沿用现有平铺世界逻辑。

## Implementation Changes
### 1. Schema And Bootstrap
新增两张表，并通过现有 `forwin/models/base.py` 轻量迁移体系创建：

- `sub_worlds`
  - `id`, `project_id`, `origin_arc_id`, `parent_subworld_id`
  - `name`, `purpose`
  - `scope`: `global_core | arc_local`
  - `status`: `active | retired`
  - `introduced_at_chapter`, `retired_at_chapter`
  - `metadata_json`
- `sub_world_roster_items`
  - `id`, `project_id`, `subworld_id`
  - `entity_id` nullable
  - `entity_kind` fixed to `character` in v1
  - `display_name`, `slot_key`, `role_hint`, `description`
  - `is_core`
  - `status`: `seeded_named | planned_slot | activated_named | retired`
  - `activation_chapter`
  - `metadata_json`

同时新增 runtime bootstrap 规则：

- 每个项目必须存在一个不可退役的 `global_core` subworld
- 迁移后对已有项目执行 bootstrap：
  - 若项目没有 `subworld`，先创建 `global_core`
  - 将已有 active character 全部挂入 `global_core`，状态为 `seeded_named`
- `ensure_active_arc_resolution()` 前先执行 `SubWorldManager.ensure_registry(project_id)`
- 如果某项目只有空的 `global_core` 且没有任何 roster，而又存在 active arc + future chapter plans，则调用 Arc 级 subworld planning 补建初始 registry

### 2. Core Manager, Repo, Updater
新增 `SubWorldManager`，集中承接下列职责：

- 汇总已有 registry 给 Arc 规划使用
- 应用 initial arc 的 `subworld` 结果
- 应用 `rearc` 的 `subworld delta`
- 为 band 计算 activation 和 chapter entry targets
- materialize roster slot 为真实 `Entity`
- 计算某章允许出场的角色名单和 context

`StateRepository` 新增查询：

- `list_subworlds(project_id)`
- `list_roster_items(project_id, subworld_ids)`
- `get_allowed_entity_names(project_id, chapter_number)`
- `get_allowed_entity_snapshots(project_id, chapter_number)`
- `get_active_subworld_summary(project_id, chapter_number)`

`StateUpdater.apply_state_changes()` 改为 strict 行为：

- 移除“找不到实体就自动 `create_entity()`”的默认路径
- 未解析到已批准角色时直接抛错
- 新增 `materialize_roster_item(...)`，只有 roster item 才能创建新人物实体

### 3. Arc Creation And Rearc Delta
`ArcDirector.plan_arc()` 扩展返回值，新增 `subworld_delta`：

- `reuse_subworld_ids`
- `retire_subworld_ids`
- `new_subworlds`
- `initial_active_subworld_ids`

`new_subworlds` 每项包含：

- `name`, `purpose`, `scope`
- `chapter_window_hint`
- `core_named_characters`
- `planned_slots`

`core_named_characters` 直接 seed 为真实 `Entity` + `seeded_named` roster item。  
`planned_slots` 只建 roster item，不建实体。

新增 `ArcDirector.plan_subworld_delta(...)`，专供 `phase3.py` 的 `rearc` 使用。输入包含：

- 项目 `premise / genre`
- 新 arc synopsis
- future chapter seeds
- 当前 `subworld` registry 摘要
- 当前 stale/focus threads

输出仍是同一套 `subworld_delta` 结构。

`phase3.py` 的 `ReplanGovernor` 改为持有 `ArcDirector`。在 `_apply_rearc()` 中：

- 创建新 `ArcPlanVersion`
- 调用 `plan_subworld_delta(...)`
- 通过 `SubWorldManager.apply_arc_delta(...)` 执行：
  - 复用已有 `subworld`
  - 新建新的 `subworld`
  - 标记可退役 `subworld`
- 然后继续现有 future chapter plan 迁移与 goal patch 逻辑

约束：

- `global_core` 允许被 reuse，但不允许被 retire
- `rearc` 只改 future chapter 所属 arc，不回写已 accepted 章节

### 4. Band Activation And Context Narrowing
复用现有 `phase24.py` 体验计划落库链，不新增 band 专用表。

扩展 `BandDelightSchedule`：

- `active_subworld_ids`
- `chapter_entry_targets`

新增 `ChapterEntryTarget`：

- `chapter_hint`
- `entity_name`
- `subworld_id`
- `role_hint`

扩展 `ChapterExperiencePlan`：

- `active_subworld_ids`
- `chapter_entry_targets`
- `entity_admission_rule = "strict_named_character"`

`phase24._persist_experience_overlay()` 在现有 schedule 生成后，调用 `SubWorldManager.plan_band_activation(...)`：

- 永远激活 `global_core`
- 再选最多 2 个 `arc_local` subworld，条件为：
  - `chapter_window_hint` 与当前 band 重叠，或
  - `purpose` / `role_hint` 能命中当前 band 的 `one_line / goals`
- 所有 active subworld 的 `is_core=true` roster item 必须 materialize
- `planned_slot` 只在被选中为 `chapter_entry_target` 时 materialize
- 每章最多 1 个 `chapter_entry_target`
- 每个 band 最多 2 个新命名角色 entry target

`assemble_context()` 与 `RetrievalBroker` 改为：

- 不再读取全项目 `get_active_entities(project_id)`
- 改为读取 `get_allowed_entity_snapshots(project_id, chapter_number)`
- `active_relations` 只保留允许实体相关边
- `allowed_entities`、`active_subworlds`、`chapter_entry_targets` 进入 `ChapterContextPack`

## Public Interfaces And Writer/Review Gates
新增协议类型：

- `SubWorldSummary`
- `SubWorldPlanDelta`
- `ChapterEntryTarget`
- `EntityMention`

扩展对象：

- `ArcDirector.plan_arc()` 返回 `subworld_delta`
- `WriterOutput` 新增 `entity_mentions`
- `BandDelightSchedule` 新增 `active_subworld_ids`、`chapter_entry_targets`
- `ChapterExperiencePlan` 新增 `active_subworld_ids`、`chapter_entry_targets`、`entity_admission_rule`
- `ChapterContextPack` 新增 `active_subworlds`、`allowed_entities`、`chapter_entry_targets`、`entity_admission_rule`
- `ProjectDetail` 新增 `subworlds`

writer 侧改造：

- prompt 明确写入硬规则：
  - 命名人物只能来自 `allowed_entities`
  - 若本章允许新人物出场，只能使用 `chapter_entry_targets`
  - 无名泛称路人允许出现，但不能固化成 canon 新人物
- 复用现有 `lore_timeline_notes_extraction`，在同一 extraction call 中加入 `entity_mentions`
- `EntityMention` 字段：
  - `entity_name`
  - `entity_kind`
  - `is_named`
  - `is_on_stage`
  - `evidence_refs`

review / canon gate：

- `ContinuityChecker` 新增 `subworld admission` 检查
- 检查来源合并：
  - `entity_mentions`
  - `state_changes`
  - `new_events`
  - `scene_outputs[].involved_entities`
- 任一命名 character 不在 `get_allowed_entity_names(project_id, chapter_number)` 中，直接报错：
  - `rule_name = "sub_world_unknown_named_entity"`
  - `severity = "error"`
  - `issue_group = "director_imbalance"`
- `_apply_canon_candidate()` 在写入前重复执行同一 admission guard
- admission guard 失败时：
  - 不写 canon
  - 记录 `CANON_COMMIT_FAILED`
  - 不再通过 `apply_state_changes()` 自动造人

### Project Detail
`build_project_detail()` 增加 `subworlds` 摘要，字段至少包含：

- `id`, `name`, `scope`, `status`
- `active_in_current_band`
- `core_cast`
- `planned_slot_count`

v1 不新增 subworld 编辑 UI；现有页面可以忽略这个新字段。

## Test Plan
必须新增或更新的测试：

- migration 后旧项目会自动拥有 `global_core`
- 旧项目已有 active characters 时，bootstrap 会把它们挂入 `global_core`
- 新项目 initial arc 会返回并 seed 非空 `subworld_delta`
- `ensure_active_arc_resolution()` 会在缺失 registry 时自动 bootstrap
- `phase3 rearc` 会调用 `plan_subworld_delta()` 并正确执行 reuse/create/retire
- `phase24` 能把 band activation 写入 `BandDelightSchedule` 和 `ChapterExperiencePlan`
- `assemble_context()` 只返回当前 chapter 允许实体
- writer prompt 包含 `allowed_entities`、`chapter_entry_targets` 和 strict 规则
- `entity_mentions` 能被 extraction 解析进 `WriterOutput`
- 未批准命名人物出现在正文、事件、state change 或 scene involved list 时，review 直接 `fail`
- canon apply 遇到未批准命名人物时不会再自动建实体
- `ProjectDetail` 返回 `subworlds` 摘要且不破坏现有字段

## Assumptions And Defaults
- 全量启用，不增加治理开关；所有项目都走新机制
- v1 只控制 `character`
- `global_core` 永远存在且不可退役
- `subworld` 只允许在两种场景创建：
  - 初始 arc planning
  - `rearc` planning
- `Band` 只激活 / materialize，不创建新的 canonical `subworld`
- `Chapter/Writer` 没有创建 `subworld` 或直接创建新命名角色的权限
