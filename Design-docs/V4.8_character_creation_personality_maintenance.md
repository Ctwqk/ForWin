# ForWin V4.8 Character Creation 与自动性格分配维护文档

更新时间：2026-04-29

状态：维护规则落档。本文档用于维护 V4.8 人物创建统一入口、自动 personality assignment、relationship-aware enrichment、coverage/backfill/reassign、World Studio 人物工作流、reviewer consistency checks 与 metrics。

V4.7 维护文档仍负责 `forwin_skills/character_personality/` 的 skill 内容、Prompt Compression 与 reusable skill 规范。V4.8 维护的是“角色如何被创建、如何自动挂载 loadout、如何被 runtime/reviewer 识别”。

## 2. 不变量

- 命名 BookState character 必须有有效 `profile.personality_loadout`。
- 自动 assignment 默认不覆盖 `manual_override=true`。
- repository 只负责持久化，不判断 personality policy。
- `CharacterCreationHelper` 是人物创建统一边界。
- `PersonalityLoadoutAssigner` 必须 deterministic、explainable、schema-validating。
- skill 只能约束行为倾向，不能发明 canon。
- writer 只拿 `active_personality_context`，不能拿完整 skill 仓库。
- reviewer evidence 使用 `personality:{character_id}`。
- stress mode 必须有 trigger。
- social mask 不能成为命名人物唯一人格机制。

## 3. 人物创建入口维护

新增或改造任何会创建命名人物的路径时，必须调用：

```python
CharacterCreationHelper.create_character(...)
CharacterCreationHelper.get_or_create_character(...)
CharacterCreationHelper.materialize_roster_character(...)
CharacterCreationHelper.import_legacy_character(...)
CharacterCreationHelper.apply_book_state_character_patch(...)
```

禁止在 helper 外新增裸调用：

```text
StateUpdater.create_entity(kind="character")
WorldNode(node_type="character") 后直接 repo.create_world_node
NodePatch(node_type="character", op="create") 后不经过 helper / integrity gate
```

允许例外：测试 fixture、schema fixture、migration dry-run fixture、`CharacterCreationHelper` 内部 legacy projection。

改动后必须跑：

```bash
uv run --extra test python -m pytest tests/test_no_bare_character_creation.py tests/test_character_creation_helper.py -q
```

## 4. Assignment Catalog 维护

维护位置：

```text
forwin_skills/character_personality/catalog/assignment_rules.yaml
forwin_skills/character_personality/catalog/compatibility_matrix.yaml
forwin_skills/character_personality/catalog/fallback_policy.yaml
forwin_skills/character_personality/catalog/character_role_taxonomy.yaml
forwin_skills/character_personality/catalog/assignment_signal_taxonomy.yaml
```

规则：

- 自动分配规则不硬编码在前端或调用点。
- `skill_id` 必须在 skill library 中真实存在。
- `skill_type` 必须与 eligible slot 匹配。
- dominant 候选必须是 `trait`。
- social mask 必须提供 `default_active_when`，除非策略明确允许默认公共面具。
- stress mode 必须提供 `default_trigger`。
- fallback skill 不能是空 skill 或占位 skill。
- 不使用 MBTI、九型、DISC、星座、临床诊断等标签作为 runtime 行为来源。

改动后必须跑：

```bash
uv run --extra test python -m pytest tests/test_personality_assignment.py tests/test_personality_runtime.py -q
```

## 5. Project Policy 维护

项目级策略读取自：

```text
Project.automation_json.character_personality
```

当前维护字段：

```json
{
  "character_personality": {
    "strict_integrity": true,
    "require_named_loadout": true,
    "low_confidence_blocks_core": false,
    "allow_background_minimal_loadout": true,
    "preserve_manual_override": true,
    "fallback_character_class": "named_supporting_character",
    "cast_diversity_enabled": true,
    "cast_diversity_adjustment": 8.0,
    "relationship_enrichment_enabled": true
  }
}
```

维护规则：

- 新增字段必须先加到 `CharacterPersonalityPolicy`。
- resolver 只读取项目配置，不接入 scoring 细节。
- policy 不得绕过 loadout schema validation。
- strict gate 调整必须考虑 legacy project 兼容。

## 6. Manual Override 维护

World Studio 或 API 手工修改 loadout 后，必须写：

```json
{
  "metadata": {
    "personality_assignment": {
      "manual_override": true,
      "assignment_mode": "manual_world_studio"
    }
  }
}
```

默认行为：

- backfill 跳过 manual override。
- reassign 默认 preserve manual override。
- relationship enrichment 不修改 manual override 角色。
- 强制覆盖必须带 reason，并写 old/new diff。

## 7. Relationship Enrichment 维护

当前自动 enrichment 只启用两个最小可运行 relationship pattern：

```text
rival / 对手 / 竞争
  -> rel-rival-respect

mentor / 导师 / 师父 / 扶持 / 保护
  -> rel-mentor-protector
```

维护规则：

- 触发点是 legacy `RelationEdge`，由 `StateUpdater.create_relation()` 调用 `RelationshipPersonalityEnricher`。
- legacy Entity 必须能通过 `WorldNode.metadata.legacy_entity_id` 映射到 BookState character id。
- enrichment 给双方补 target-specific `relationship_patterns`。
- 不覆盖 manual override。
- 不重复添加同 `target + skill` pattern。
- 每次产生 diff 都写 `PERSONALITY_RELATIONSHIP_ENRICHED`。
- 自动选择新 relationship skill 前，必须先把该 skill 从占位内容补成可运行内容。

改动后必须跑：

```bash
uv run --extra test python -m pytest tests/test_character_relationship_enrichment.py tests/test_character_personality_integration.py -q
```

## 8. API 维护

V4.8 相关 API：

```text
POST /api/projects/{project_id}/characters
POST /api/projects/{project_id}/characters/personality/preview
POST /api/projects/{project_id}/characters/personality/active-context/preview
GET  /api/projects/{project_id}/characters/personality/coverage
GET  /api/projects/{project_id}/characters/personality/metrics
POST /api/projects/{project_id}/characters/personality/backfill
POST /api/projects/{project_id}/characters/personality/relationships/enrich
POST /api/projects/{project_id}/characters/{character_id}/personality/reassign
GET  /api/projects/{project_id}/characters/{character_id}/personality/assignment-report
GET  /api/projects/{project_id}/characters/personality/assignment-reports/{assignment_id}
PUT  /api/projects/{project_id}/book-state/characters/{character_id}/personality-loadout
```

维护规则：

- API 层只组装请求和错误码，不复制 assignment 策略。
- explicit invalid loadout 应返回 400。
- assignment report by id 应从 BookState metadata 与 DecisionEvent 追溯。
- coverage/metrics 不能改变数据库状态。
- backfill/reassign/enrich 产生数据库变更时必须写 DecisionEvent。

## 9. World Studio 维护

World Studio 人物性格页当前包含：

- coverage summary 与 filters。
- metrics summary。
- 人物创建表单。
- 自动 personality preview。
- 可编辑 loadout JSON。
- assignment report view。
- reassign。
- active context preview。
- relationship enrichment。
- skill catalog 摘要。

维护规则：

- 前端不得硬编码 assignment scoring 规则。
- skill 列表从 `/api/personality-skills` 读取。
- 创建人物时，如果用户编辑 loadout JSON，则发送 `personality_loadout` 和 `personality_policy="manual"`。
- active context preview 调后端 API，不在前端重建 runtime。
- 页面文案必须避免把 reference model 当运行时行为来源。

涉及前端时必须跑：

```bash
uv run --extra test python -m pytest tests/test_world_studio_frontend.py -q
cd frontend/world-studio && npm run build
```

## 10. Reviewer 维护

`PersonalityConsistencyReviewer` 当前做 deterministic 可检测子集：

- `personality_missing_loadout`
- `reference_model_override`
- `stress_mode_without_trigger`

维护规则：

- reviewer issue 或 lint signal 必须引用 `personality:{character_id}`。
- reviewer 只能使用 `active_personality_contexts`、integrity issues、writer output 可见文本和现有 review context。
- 不直接读取完整 skill body 推导判罚。
- LLM-based OOC 深度归因不属于 deterministic reviewer 的职责。

新增 reviewer 检查后必须跑：

```bash
uv run --extra test python -m pytest tests/test_reviewer_personality_consistency.py tests/test_personality_runtime.py -q
```

- low confidence / fallback / manual override 以 `WorldNode.metadata.personality_assignment` 为准。
- missing loadout 以 BookState character profile 为准。

## 12. Troubleshooting

### 新角色没有 loadout

检查顺序：

1. 创建路径是否经过 `CharacterCreationHelper`。
2. `CharacterCreationRequest.personality_policy` 是否错误设置。
3. explicit loadout 是否 schema invalid。
4. fallback skill 是否真实存在且不是占位。
5. `tests/test_no_bare_character_creation.py` 是否覆盖该入口。

### 自动分配结果不稳定

检查：

- catalog 是否有同分候选。
- tie-break 是否仍使用 seeded rank。
- 是否引入非 deterministic random。
- cast diversity 是否依赖无序输入。

### manual override 被改掉

检查：

- `metadata.personality_assignment.manual_override` 是否为 true。
- backfill request 是否显式强制覆盖。
- reassign request 是否 `force=true` 且有 reason。
- relationship enrichment 是否跳过 locked node。

### relationship pattern 没进入 active context

检查：

- loadout 中 relationship pattern 是否有 `target`。
- context assembler / preview 是否传入 `relationship_targets`。
- skill 是否存在且不是占位。
- 当前场景是否真的包含该 target relationship。

## 14. 部署注意

本增量涉及后端 API、数据库读写路径、World Studio 前端和 reviewer。部署前：

1. 确认没有 active generation task。
2. 重建后端与 World Studio 前端。
3. 对一个测试项目执行 coverage、metrics、preview、active context preview。
4. 对 legacy 项目先 dry-run backfill，再执行实际 backfill。
5. 观察 DecisionEvent 中是否出现 expected assignment / reassign / enrichment / integrity events。
