# ForWin V3.0 WorldModel 设计规格

更新时间：2026-04-24

来源：`V2_9_3_skill_runtime.md` 当前主干架构 + V3 WorldModel 分轨计划 + 当前 `codex/world-model-v3` 实现。

说明：本文件定义 ForWin V3.0 的 WorldModel 路线。V3.0 不替代现有 DB、Genesis、StateUpdater、DecisionEvent 或 reviewer 主链，而是在章节 canon 之后新增一个确定性编译层，把小说世界当前状态投影成可版本化、可检索、可导出的 World Wiki。

---

## 1. 核心结论

ForWin V3.0 的核心升级是：

```text
DB / raw events / DecisionEvent
        ↓
WorldModelCompiler
        ↓
WorldModelSnapshot / Page / Link / Conflict
        ↓
writer / reviewer context
        ↓
Obsidian Vault / World Studio
```

权威关系固定为：

- 数据库里的项目、Genesis、章节、事件、状态、线程、时间线、DecisionEvent 仍是源。
- `WorldModelSnapshot` 是截至某章后的可重建世界状态投影。
- Markdown / Obsidian 是人类可读投影，不是 canon。
- Obsidian 修改只进入 `WorldEditProposal`，不能直接改 DB canon。
- LLM 后续只能用于页面润色、Open Questions、Contradiction explanation 或 proposal 建议，不拥有 canon 写权。

---

## 2. V3.0 相对 V2.9.3 的版本边界

`V2.9.3` 已完成：

- Genesis 根层前置化与 `BookGenesisPack.world`
- Writer / Reviewer / Skill Runtime / PromptTrace
- accepted chapter 后的 state/events/thread/timeline/phase3/phase4 后置产物
- DecisionEvent 治理链
- 原生 JS 首页与任务/Genesis 工作台

`V3.0` 新增：

- `forwin/world_model/` 编译层
- `world_model_*` 数据表与轻量迁移
- Genesis -> 第 0 章 WorldModel bootstrap
- accepted chapter -> post-acceptance WorldModel compile
- `WorldContextPack` 接入 writer / reviewer context
- Obsidian Markdown / Canvas 只读导出
- Obsidian import -> proposal，不直接写 canon
- React/Vite World Studio，只承载 WorldModel 管理端
- MCP 只读 world model 工具

本轮明确不做：

- 不把 Obsidian 当主数据库
- 不把旧首页整体迁移到 React
- 不让 LLM 自主改 canon
- 不在首版实现完整 proposal adapter 到所有 canon 表的映射
- 不把 conflict detector 的结果自动修剧情

## 3. 四阶段路线图

原 V3.1-V3.6 六轨路线已经压缩为 V3.1-V3.4 四个交付阶段。V3.0 本身视为 WorldModel 方向的总设计和当前基线，不再理解为“已经完整实现后还剩六个阶段”。当前代码已经落下 V3.1 的主体，以及 V3.2/V3.3/V3.4 的部分骨架；后续按四阶段补齐即可。

合并关系：

| 新阶段 | 合并来源 | 阶段目标 |
| --- | --- | --- |
| `V3.1` | 原 V3.1 | WorldModel 核心基座、schema/store/compiler、只读 Obsidian 导出 |
| `V3.2` | 原 V3.2 + 原 V3.3 | accepted 后自动编译，并真正进入 writer / reviewer context |
| `V3.3` | 原 V3.4 + 原 V3.5 | React World Studio + Obsidian proposal 工作流 |
| `V3.4` | 原 V3.6 + 后续 hardening | 冲突治理、Canvas 图谱、LLM Wiki 润色、governance 接入 |

### 3.1 V3.1：WorldModel 核心基座与只读 Wiki

目标：建立一个可重建、可版本化、可导出的世界模型基座。此阶段的交付标准是“能从 Genesis 和已有 canon 状态编译出 WorldModelSnapshot，并导出 Obsidian 可读 Wiki”，不处理 Obsidian 回写。

范围：

- 新增 `forwin/world_model/` 核心模块。
- 新增 `forwin/protocol/world_model.py` 或等价 schema 层。
- 新增 `forwin/models/world_model.py` 和轻量迁移。
- 实现 `WorldModelCompiler.bootstrap_from_genesis()`。
- 实现 `WorldModelStore` 的 snapshot/page/link/conflict/compile run 持久化。
- 实现只读 Obsidian export。
- 输出 `00_Index.md`、`AGENTS.md`、`world_model_schema.md`、角色页、势力页、地图页、制度/资源页、秘密/承诺页、冲突页。
- 基础 Canvas 可以在本阶段作为只读投影落地，但不要求图谱关系完整。

数据表：

- `world_model_snapshots`
- `world_model_pages`
- `world_model_links`
- `world_model_conflicts`
- `world_edit_proposals`
- `world_model_compile_runs`

关键接口：

```python
WorldModelCompiler.bootstrap_from_genesis(project_id: str) -> WorldModelSnapshot
WorldModelStore.latest_snapshot(project_id: str, as_of_chapter: int | None = None)
ObsidianWorldExporter.export_project(project_id: str, vault_root: Path | None = None)
```

验收标准：

- 空库和旧库都能创建新表。
- locked Genesis 能生成 `as_of_chapter=0` snapshot。
- snapshot 覆盖 `world_root / space_model / actor_model / institution_model / economy_model / time_model / technology_model / plot_model / quality_model` 的最小结构。
- Obsidian vault 默认导出到 `data/world_vaults/{project_id}/`。
- Markdown page 带 frontmatter、locked sections、Manual Notes。
- 导出不会修改 canon。

当前状态：

- 主体已实现。
- `schema.py` 实际落在 `forwin/protocol/world_model.py`。
- 当前 compiler 是 deterministic snapshot compiler；还不是独立 delta compiler。
- 章节页、完整 systems 页、精细 Canvas 仍需后续补强。

### 3.2 V3.2：Canon 后自动编译与上下文接入

目标：让 WorldModel 从“只读百科投影”升级为写作链的一部分。此阶段的交付标准是“每章 accepted 后生成新 snapshot，writer / reviewer 在下一章能读到最新成功世界状态”。

范围：

- 在自动生成链路和人工 `accept_review()` 链路接入 post-acceptance compile。
- 触发点放在 canon 写入、phase3、phase4 后置产物之后。
- 编译失败不回滚已接受 canon，但要记录 failed compile run 和 DecisionEvent。
- 自动生成遇到严重编译失败时暂停后续章节。
- `compile_after_chapter(project_id, chapter_number)` 必须按 `source_digest` 幂等。
- 新增 / 完善 `WorldModelRetriever`。
- `ChapterContextPack` 和 `ReviewContextPack` 挂 `WorldContextPack`。
- `RetrievalBroker` 在预算不足时保留 hard constraints / conflicts / secrets，最后裁剪 overview。
- writer prompt 和 reviewer evidence index 增加 world refs。

核心流程：

```text
chapter accepted
  -> StateUpdater 写 state/events/thread/timeline
  -> phase3 / phase4 写 stage analysis、world pressure、NPC intents
  -> WorldModelCompiler.compile_after_chapter()
  -> new snapshot/pages/conflicts
  -> next writer/reviewer context reads latest successful snapshot
```

验收标准：

- accepted chapter 后产生新 live snapshot。
- 重跑同一章、同一 source digest 不产生重复 live snapshot。
- 编译失败写入 `world_model_compile_runs(status=failed)`。
- 编译失败写入 `WORLD_MODEL_COMPILE_FAILED` DecisionEvent。
- writer context 包含 relevant world pages。
- reviewer evidence index 包含 world model refs。
- legacy 非 Genesis 项目不会因缺少 Genesis revision 导致旧回归失败。

当前状态：

- 自动编译、幂等、compile run、DecisionEvent、legacy fallback 已有骨架。
- writer/reviewer context 已接入。
- 仍需补强 page selection、secret visibility、active constraints 的精细排序。

### 3.3 V3.3：World Studio 与 Obsidian Proposal 闭环

目标：提供人类可用的世界模型管理端，并把 Obsidian 修改纳入 proposal / review 流程。此阶段的交付标准是“人能在 World Studio 浏览世界、导出/导入 Obsidian、审核 proposal，但不会直接污染 canon”。

范围：

- 新增 `frontend/world-studio/`，使用 TypeScript + React + Vite + lucide-react。
- FastAPI 新增 `/world-studio` 静态入口。
- 旧首页只增加打开 World Studio 的入口，不迁移任务中心。
- API 统一放在 `/api/projects/{project_id}/world-model/...`。
- World Studio 首版包含：
  - project selector
  - snapshot timeline
  - world pages list/detail
  - character/faction/region detail
  - conflict list
  - Obsidian export/import
  - proposal list
  - proposal accept/reject
- `importer_obsidian.py` 扫描 vault content hash，生成 `WorldEditProposal`。
- import 不直接改 `world_model_pages.markdown`，不直接改 EntityState / RelationEdge / Genesis。
- proposal review 状态支持 `accepted / rejected / superseded`。

后续 adapter 边界：

- `Genesis patch`
- `NarrativeConstraint`
- `EntityState amendment`
- `RelationEdge update`
- `Conflict resolution`

无法映射的 proposal 保留为人工待办。

验收标准：

- React build 通过。
- `/world-studio` 可加载构建产物。
- World Studio 能加载项目、pages、conflicts、proposals。
- export 按钮能生成 vault。
- import 只生成 proposal。
- proposal review 状态流正确。
- accepted proposal 触发重新编译框架。
- Obsidian 修改不会直接改 canon。

当前状态：

- World Studio 只读首版已实现。
- export/import/proposal review 已有骨架。
- diff viewer、protected section 级 diff、adapter 写回 canon 尚未完成。

### 3.4 V3.4：冲突治理、图谱与 LLM Wiki 维护

目标：把 WorldModel 从“可读状态投影”升级为“长期维护小说世界一致性的治理系统”。此阶段的交付标准是“系统能稳定发现关键世界矛盾、图谱可用、LLM 只做 Wiki 维护和 proposal 建议，不越权写 canon”。

范围：

- 扩展 `conflict_detector.py` 的确定性规则：
  - `dead_alive_conflict`
  - `character_location_conflict`
  - `secret_revealed_too_early`
  - `unintroduced_entity`
  - `map_node_missing`
  - `faction_control_conflict`
  - `timeline_order_conflict`
  - `resource_logic_conflict`
  - `promise_untracked`
- World Studio 增加 conflict detail、resolution proposal、severity filter。
- Obsidian Canvas 自动生成并持续更新：
  - `character_relationships.canvas`
  - `faction_conflicts.canvas`
  - `map_overview.canvas`
  - `secret_reveal_ladder.canvas`
  - `arc_dependencies.canvas`
- 引入 LLM Wiki maintenance，但权限固定为：
  - 页面润色
  - Open Questions
  - Contradiction explanation
  - proposal drafting
  - compile log summary
- LLM 不允许：
  - 删除 canon
  - 改角色死亡状态
  - 改势力归属
  - 提前揭示秘密
  - 强行修改未来计划
- 可选把严重 world conflict 映射为 governance blocking reason；默认先展示和暂停，不自动改剧情。

验收标准：

- conflict detector 每类规则至少有一个确定性测试。
- conflict rows 能稳定生成、关闭或 supersede。
- Canvas 文件能被 Obsidian 打开，并有稳定 node / edge。
- LLM Wiki maintenance 输出只进入 page/proposal/review note，不直接写 canon。
- severe conflict 能在 World Studio 明确展示，并可触发人工处理流程。

当前状态：

- 基础 conflict rows、contradiction pages、基础 Canvas 已有。
- 完整规则集、治理阻断映射、LLM Wiki maintenance 尚未实现。

## 4. 数据模型

### 4.1 `world_model_snapshots`

用途：保存每次编译后的完整世界状态。

核心字段：

- `project_id`
- `as_of_chapter`
- `version`
- `status`
- `snapshot_json`
- `source_digest`
- `compiled_from_event_id`

写作第 `N+1` 章前，writer / reviewer 默认读取 `as_of_chapter <= N` 的最新 live snapshot。

### 4.2 `world_model_pages`

用途：对应 Obsidian / World Studio 中的一个可读页面。

核心字段：

- `page_key`
- `page_type`
- `title`
- `vault_path`
- `markdown`
- `frontmatter_json`
- `content_hash`
- `revision`
- `status`
- `as_of_chapter`

`page_type` 首批包含：

- `overview`
- `character`
- `faction`
- `region`
- `node`
- `institution`
- `resource`
- `secret`
- `promise`
- `contradiction`

### 4.3 `world_model_links`

用途：保存页面之间的图关系。

首批来源：

- RelationEdge 派生的人物 / 势力关系
- 后续可接角色归属、势力控制、地图连接、秘密依赖、arc 依赖

### 4.4 `world_model_conflicts`

用途：保存确定性矛盾或风险。

字段包含：

- `conflict_type`
- `severity`
- `subject_key`
- `description`
- `evidence_refs_json`
- `status`

### 4.5 `world_edit_proposals`

用途：承接 Obsidian / World Studio 的人工修改建议。

状态：

- `pending`
- `accepted`
- `rejected`
- `merged`
- `superseded`

### 4.6 `world_model_compile_runs`

用途：记录每次编译尝试，便于追踪、失败诊断和重跑。

字段包含：

- `trigger`
- `source_refs_json`
- `source_digest`
- `status`
- `snapshot_id`
- `error`

```yaml
---
forwin_id: character:lin-jin
project_id: xxx
page_type: character
status: canon_live

snapshot_id: xxx
source_refs:
  - book_genesis_revision:xxx
```

正文固定分区：

- `Current State`

- locked sections 的修改只生成 proposal。
- Markdown 不反向覆盖 canon。

## 12. 设计原则

V3.0 后 ForWin 的核心能力从：

```text
生成章节
```

升级为：

```text
维护一个会随章节推进而变化的小说世界状态机，
并让 writer / reviewer / 人类编辑都围绕这个世界模型工作。
```

但根原则不变：

- canon 来自 DB，不来自 Markdown
- LLM 维护投影，不越权改真相
- 世界模型按章节版本化
- 所有修改要有 evidence / proposal / decision trace
- 世界模型必须服务写作，而不只是百科
## 4.1 Amendment

`provisional rehearsal` 不再作为 3.0 核心机制推进。3.0 的前置验证入口统一改为 `Scenario Rehearsal`：检查 chapter/band contract、future constraints、visibility/cognition guard 和 plan patch/replan 入口；正文产物层使用 `Candidate Draft Review` 命名承接原有 draft/review/repair/canon 流。完整 4.1 补齐后，Candidate Draft 记录层显式追踪 candidate body、scene/state/event/thread candidates、review/repair/canon 状态。
